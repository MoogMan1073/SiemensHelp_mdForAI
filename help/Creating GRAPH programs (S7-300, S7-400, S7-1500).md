### GRAPH programming language (S7-300, S7-400, S7-1500)

#### Overview

GRAPH is a graphic programming language for creating sequential control systems. Sequences can be programmed in a fast straightforward manner using sequencers. The process is broken down into individual steps, each with a clear scope of functions, and organized into sequencers. The actions to be executed are defined within the individual steps. There are transitions between the steps. These contain conditions for advancing to the next step.

#### Sequential control system modules

A sequential control system controls the process in a pre-defined order and subject to certain conditions. The complexity of the sequential control system is specified by the automation task. At least three blocks always belong to a sequential control system.

- GRAPH function block

  In the GRAPH function block, you describe the individual steps and transitions of the sequential control system within one or more sequencers.
- Instance data block

  The instance DB contains the data and parameters of the sequential control system. It is assigned to the GRAPH function block and can be automatically generated by the system.
- Calling code block

  In order for the GRAPH function block to be executed within the cycle, it must be called from a higher-level code block. This block can be an organization block (OB), a function (FC), or another function block (FB). The GRAPH function block is always called as a single instance.

The following figure shows how the blocks work together in a sequential control system:

![Sequential control system modules](images/49070382603_DV_resource.Stream@PNG-en-US.png)

① Calling block

② GRAPH function block

③ Instance data block

The cycle of a GRAPH function block is defined by the cycle of the calling block. In each cycle, the permanent pre-instructions are executed first in the GRAPH function block. Then the actions of the active steps are processed. Last, the permanent post-instructions are executed.

> **Note**
>
> Note the following:
>
> - Permanent instructions are executed in each cycle, even if no step is active.
> - Due to their functionality, GRAPH blocks require a longer processing time in the CPU. This is due to the following factors:
>
>   - Additional implicit diagnostics
>
>   - Integrated coordination of the process
>
>   - Implemented operating modes for sequencer processing

#### Basics on sequencers (S7-300, S7-400, S7-1500)

##### Introduction

Within a GRAPH function block, you can write your program in the form of sequencers. The sequencers can either process independent tasks, or you can break down a complex task into multiple sequencers. If the sequencers are processing independent tasks, they are processed in parallel in the program flow. If you are using multiple sequencers to break down a complex task into smaller segments, you must link the associated sequencers with jumps.

You subdivide the program into individual steps within a sequencer. In the simplest case, the steps are processed linearly one after the other. However, the use of alternative or simultaneous branches allows you to create sequencers with more complex structures. The program execution always begins with a step that you have defined as the initial step. A sequencer can have one or more initial steps. The initial steps can be at any position in the sequencer.

When a step is active, its actions are executed. More than one step can be be active simultaneously, e.g., within a simultaneous branch. While a step is active, the conditions of the following transition are constantly checked. As soon as all conditions are satisfied and no supervision errors are pending, the transition advances to the next step. This now becomes the active step. You can complete a sequencer either with a jump or a sequence end. The destination of the jump can be any step of the same sequencer or another sequencer. This enables cyclic processing of the sequencer.

##### Processing principle of a sequencer

The processing of a sequencer begins when the initial step(s) of the sequence become active. Several initial steps are permitted only in simultaneous branches. As long as a step is active, its actions are executed. The interlock conditions for the individual actions are taken into consideration here. Once all possible actions have been executed, a check is first made for the existence of a supervision error. If no such error exists and the following transition condition is satisfied, the next step of the sequence becomes active. If a supervision error exists or the transition condition is not satisfied, the step remains active until the error is eliminated or the transition condition is satisfied. At the end of a sequencer, you can use jumps to enable cyclic processing of the sequencer or a sequence end to terminate the sequencer.

##### **Sequencers and temporary variables**

Before executing a sequencer, the conditions of the subsequent transitions are checked. At this time, existing temporary variables are in the initial state. Afterwards the two active steps determined before are executed. If a temporary variable is changed within a step, which is used in one of the following transitions, this change does not affect the further switching of the transition. The reason for this is that the variable has already been evaluated before or has the initial value again in the next cycle. Therefore, use static instead of temporary variables in the step enabling conditions of the sequencers.

##### Simultaneous branch

You can use simultaneous branches to program AND branches. This means that you use one transition to activate multiple steps whose actions are then executed. Simultaneous branches always start with a step.

The subsequent transitions of the simultaneous branches are located on the main branch, in which case the various simultaneous branches can be connected to various points of the main branch. Note that branches that you join together in a transition will only advance to the next step when all branches are completely processed.

You can program up to 249 simultaneous branches in a sequencer.

The following figure shows the graphic representation of a simultaneous branch:

![Simultaneous branch](images/52607221259_DV_resource.Stream@PNG-de-DE.png)

##### Alternative branch

You can use alternative branches to program OR branches. This means that you insert branches that start with a transition after a step. Depending of which transition's condition is satisfied first, the corresponding branch is executed. If multiple transitions are fulfilled at the same time, the set operating mode determines which branch is executed:

- Automatic or semi-automatic mode: The transition furthest to the left receives the highest priority and the corresponding branch is executed.
- Manual mode: The transition that is fulfilled first receives the highest priority and the corresponding branch is executed.

Should, however, the "Process all transitions" test setting be activated, then the branch with the lowest transition number will be processed first in both cases.

You can program up to 125 alternative branches in a sequencer.

The following figure shows the graphic representation of an alternative branch:

![Alternative branch](images/52607592075_DV_resource.Stream@PNG-de-DE.png)

##### Close branch

You can use the "Close branch" element to close simultaneous and alternative branches to their parent branch. This is necessary if you do not want to complete the branch with a jump or a sequence end.

##### Jump to step

You can use a jump to continue the program execution at any step within the GRAPH function block. Jumps can be inserted at the end of the main branch, an alternative branch, or a simultaneous branch to enable cyclic processing of the sequencer. The jump and jump destination are represented in the sequencer as arrows, in which case the return transition is specified for the jump destination and the destination step is specified for the return.

> **Note**
>
> Avoid jumps from a transition to a directly preceding step. If you need this type of jump, insert an empty step with a transition without condition.

The following figure shows the graphic representation of a jump:

![Jump to step](images/52608735371_DV_resource.Stream@PNG-de-DE.png)

##### Sequence end

You can complete the sequencer or a branch with the "Sequence end" element. This terminates the sequencer or the branch. In a simultaneous branch, a transition must precede the sequence end. The "Sequence end" element is represented in the sequencer with a black dot.

The following figure shows the graphic representation of a sequence end:

![Sequence end](images/52608560523_DV_resource.Stream@PNG-de-DE.png)

> **Note**
>
> If all branches of a sequencer are completed with a sequence end, the sequencer can be restarted by the "INIT_SQ" parameter or using the "Initialize" button in the "Sequence control" pane of the "Testing" task card.

#### Basics on steps (S7-300, S7-400, S7-1500)

##### Introduction

Steps are used to subdivide complex automation tasks into straightforward subtasks that can then be accomplished using actions. The individual steps are organized in sequencers so that every step can be executed in the specified order in the program flow. Each step must be assigned a unique name and number.

The actions of a step are carried out when the step becomes active. For this a step must either be an initial step or the transition condition preceding it must be satisfied. A step can also be called by an event-dependent action. In addition, you can program interlock and supervision conditions in a step.

You are free to choose the step at which the processing of the sequencer is to start. For this you specify any step as the initial step. In simultaneous branches you can specify multiple steps as initial steps. During cyclic operation of a sequencer, the initial step is treated like every other step, provided there is no initialization of the sequencer using the "INIT_SQ" parameter. If the sequencer is reinitialized, all initial steps are activated when there is a positive signal edge at "INIT_SQ". All other steps are deactivated. Initial steps are framed in the navigation view and sequence view to make them easy to recognize.

A step in which no actions are programmed is an empty step. Since an empty step does not have to process an action, the sequencer advances immediately to the following transition. This allows you to program two consecutive transitions.

You can change the name and number of a step at any time. In addition, you have the option of documenting steps using comments.

#### Step elements (S7-300, S7-400, S7-1500)

You can program the following components in a step:

- Interlock
- Supervision
- Actions

##### Interlock

You use an interlock to program conditions on which the execution of the individual actions depends. Actions of a step that are linked with the interlock are only executed when the interlock condition is satisfied. If the interlock condition is not satisfied, a fault exists. You can specify alarms for this case, which are then displayed. The fault does not affect the advancement to the next step, however. Note also that the interlock is automatically canceled when the step is deactivated. Optionally, you can specify that the operator must acknowledge an interlock alarm in order to close it.

You also have the option to not program any conditions in the interlock but to still link the interlock with actions. In this case, the interlock condition is always satisfied.

##### Supervision

You use a supervision to program conditions with which you can monitor a whole step. If the condition is satisfied, a fault exists and the step remains active. That is, in this case the sequencer does not advance to the next step but instead remains in the active step until the fault has been eliminated. You can also specify alarms and mandatory acknowledgement for the supervision.

##### Actions

Actions are programmed within the steps. For example, actions control inputs or outputs, activate or deactivate other steps of the sequencer, or call blocks. Actions thus contain instructions for process control.

Actions are differentiated into standard actions and event-dependent actions. Standard actions are executed as long as the step is active. Event-dependent actions are those that you have linked with an event.

#### Basics on transitions (S7-300, S7-400, S7-1500)

##### Introduction

Transitions are located between the steps and contain the conditions for advancing to the next step. That is, the sequencer only switches to the subsequent steps when the conditions of the transition are satisfied. In the process, the steps belonging to the transition are deactivated and the subsequent steps are activated. If the conditions are not satisfied, the steps belonging to the transition remain active. Each transition must be assigned a unique name and number. You can program the conditions of a transition in LAD or FBD.

A transition in which no conditions are programmed is an empty transition. In this case, the sequencer advances immediately to the subsequent steps.

You can change the name and number of a transition.

#### Basics on actions (S7-300, S7-400, S7-1500)

##### Introduction

You can use actions, for example, to control inputs or outputs, activate or deactivate other steps of the sequencer, or call blocks. Actions are executed when the associated step becomes active (standard actions) or when an event linked with the action occurs (event-dependent actions). You can also use timers and counters in actions.

The following table shows examples of programming of actions:

| Interlock | Event | Identifier | Action | Description |
| --- | --- | --- | --- | --- |
|  |  | N | MyTag1 | Action consisting of instruction and operand without interlock |
| -(C)- |  | N | MyTag2 | Action consisting of instruction and operand with interlock |
|  | S1 | S | MyTag3 | Action consisting of event, instruction, and operand without interlock |

Actions are executed in order from "top" to "bottom" when the step is active or the event occurs.

The following figure shows an example of the order in which actions are processed in a step:

![Introduction](images/50016103691_DV_resource.Stream@PNG-en-US.png)

① Interlock

② Optional event

③ Identifier

④ Operand, assignment, or block call

An action consists of the following components:

- Interlock (optional)

  You can link an action with an interlock in order to influence the execution of the action.
- Event (optional)

  An event defines when an action is to be executed. You must specify an event for some identifiers.
- Identifier (mandatory)

  The identifier defines the type of action to be executed, e.g., the setting or resetting of an operand.
- Action (mandatory)

  The action determines the operands used to execute the action.

#### Standard actions (S7-300, S7-400, S7-1500)

##### Introduction

Standard actions are executed as long as a step is active. The following table shows the possible standard actions and their identifiers:

| Identifier | Data type of the operand | Meaning |
| --- | --- | --- |
| N - Set as long as step is active | BOOL  FB, FC, SFB, SFC | The signal state of the operand is "1" as long as the step is active.  The specified block is called as long as the step is active.  The step also counts as being active in the cycle in which the S1 event occurs. |
| S - Set to 1 | BOOL | As soon as the step is active, the operand is set to "1" and then remains at "1". |
| R - Set to 0 | BOOL | As soon as the step is active, the operand is set to "0" and then remains at "0". |
| D - On delay | BOOL and Time/DWORD | n seconds after the step activation, the operand is set to "1" and remains at "1" for the duration of the step activation. This does not apply if the duration of the step activation is shorter than n seconds.  You can specify the time as a constant or as a PLC tag of data type TIME/DWORD. |
| L - Set for limited time | BOOL and Time/DWORD | When the step is active, the operand is set to "1" for n seconds. The operand is then reset. The operand is also reset if the step activation time is less than the time duration.  You can specify the time as a constant or as a PLC tag of data type TIME/DWORD. |

You can link all of these standard actions with an interlock. If an action is linked with an interlock, the action is only executed if the conditions of the interlock are satisfied.

The following table shows some examples of the use of time constants:

| Identifier | Action | Description |
| --- | --- | --- |
| D | "MyTag", T#2s | 2 seconds after step activation, the "MyTag" operand is set to "1" and remains at "1" for the duration of the step activation. This does not apply if the duration of the step activation is shorter than 2 seconds. The operand is reset with step deactivation (not latching). |
| L | "MyTag", T#20s | If the step is active, the "MyTag" operand is set to "1" for 20 seconds. The operand is then reset (not latching). The operand is also reset if the step activation time is less than 20 seconds. |

#### Event-dependent actions (S7-300, S7-400, S7-1500)

##### Introduction

You have the option of linking an action with an event in order to make the execution of the action dependent on certain conditions. For actions with identifiers "D", "L", and "TF", the linking with events is not possible. If an action is linked with an event, the signal state of the event is detected by edge evaluation. This means that the instructions are only executed in the cycle in which the event occurs.

The following table shows the events that you can link with actions:

| Event | Signal evaluation | Description |
| --- | --- | --- |
| S1 | Positive edge | Step is activated (signal state = "1") |
| S0 | Negative edge | Step is deactivated (signal state = "0") |
| V1 | Positive edge | Supervision condition is satisfied, i.e., fault occurs (signal state = "1") |
| V0 | Negative edge | Supervision condition is no longer satisfied, i.e., fault is eliminated (signal state = "0") |
| L0 | Positive edge | Interlock condition is satisfied, i.e., fault is eliminated (signal state = "1") |
| L1 | Negative edge | Interlock condition is not satisfied, i.e., fault occurs (signal state = "0") |
| A1 | Positive edge | An alarm is acknowledged. |
| R1 | Positive edge | Incoming registration. |

The signal states of an event can be used to program additional actions. This allows you to monitor and influence not only the individual steps but also the entire sequential control system.

Actions for which you use the identifiers "ON" (activate step) and "OFF" (deactivate step) must always be linked with an event. The event determines the timing of the activation or deactivation of the step. If a step is both activated and deactivated in a cycle, the deactivation has higher priority.

You can additionally connect actions for which you use the "S1", "V1", "A1", or "R1" events with an interlock. As a result of this, the actions are only executed if the conditions of the interlock are satisfied.

The following figure shows an example of a step that contains event-dependent actions:

![Introduction](images/50149634827_DV_resource.Stream@PNG-en-US.png)

As soon as the "Step4" step becomes active and the interlock condition is satisfied, "MyTag1" is set to "0" and then remains at "0". If a supervision error occurs, the "MyStep" step is deactivated. If an interlock error occurs, all steps are deactivated.

##### Registration

A registration is an event that is triggered outside the block. Registrations are queried via a positive signal edge at the "REG_S" or "REG_EF" input parameter. If the registration occurs via the "REG_S" input parameter, the event is only forwarded to the active step that is indicated in the "S_NO" output parameter. If the registration is via the "REG_EF" input parameter, the event is forwarded to all currently active steps.

#### Block calls (S7-300, S7-400, S7-1500)

##### Introduction

Within actions you can call other function blocks and functions in which you perform certain subtasks. This allows you to better structure your program.

##### Call of function blocks (FB)

Use the following syntax for calling a function block:

CALL "<FBName>", "<DBName>" (list of parameters)

The following table shows an example of the call of a function block:

| Interlock | Event | Identifier | Action |
| --- | --- | --- | --- |
|  |  | N | CALL "FB_KOP", "FB_KOP_DB" (MyInOut := "MyTag2") |

The "FB_KOP" function block is called using the "CALL" instruction. The "CALL" instruction is always connected with the "N" identifier. After the name of the called block is the name of the data block containing the data and parameters of the function block. In the list of parameters, the "MyTag2" tag is assigned to the "MyInOut" parameter.

##### Call of functions (FC)

Use the following syntax for the call of a function:

CALL "<FCName>" (list of parameters)

The following table shows an example of the call of a function:

| Interlock | Event | Identifier | Action |
| --- | --- | --- | --- |
|  |  | N | CALL "FC_KOP" (MyInput := "MyTag1") |

The "FC_KOP" function is called using the "CALL" instruction. In the list of parameters, the "MyTag1" tag is assigned to the "MyInput" parameter.

#### Timers (S7-300, S7-400, S7-1500)

##### Introduction

You can use timers within actions. Except for "TF", all timers are dependent on an event that determines when the timer is activated. The "TF" timer is activated by the step itself.

When the "TL", "TD", and "TF" timers are used, you must specify the duration. Use the following syntax for this:

<Duration> = IWy, QWy, MWy, LWy, DBWy, DIWy; tag of type S5TIME or WORD; S5T#Constant

y = 0 ... 65534

You can also enter the constants as normal time information, such as "5 s".

The following table shows the timers you can use in actions:

| Event | Identifier | Description |
| --- | --- | --- |
| S1, S0, L1, L0, V1, V0, A1, R1 | TL | Extended pulse:  As soon as the defined event occurs, the timer starts. During the specified duration, the timer status has signal state "1". After the time expires, the timer status has signal state "0". |
| S1, S0, L1, L0, V1, V0, A1, R1 | TD | Retentive on-delay:  As soon as the defined event occurs, the timer starts. During the specified duration, the timer status has signal state "0". After the time expires, the timer status has signal state "1". |
| S1, S0, L1, L0, V1, V0, A1, R1 | TR | Stop timer and reset:  As soon as the defined event occurs, the timer is stopped. Timer status and time value are reset to 0. |
| - | TF | Off delay:  The counter status is set to "1" as soon as the step is activated. The timer starts running when the step is deactivated, but the timer status is reset to "0" only after the time expires. |

#### Counter operations (S7-300, S7-400, S7-1500)

##### Introduction

You can use counters within actions. Counters always require an event in order to specify when the counter is activated. This means that the counter is activated when the associated event occurs. You can additionally connect actions for which you use the S1, V1, A1, or R1 events with an interlock. As a result of this, the actions are only executed if the conditions of the interlock are satisfied.

The following table shows the counters you can use in actions:

| Event | Identifier | Data type of the operand | Meaning |
| --- | --- | --- | --- |
| S1, S0, L1, L0, V1, V0, A1, R1 | CS | COUNTER | Set initial counter value:   As soon as the defined event occurs, the counter is set to the specified count value. You can specify the counter value as a tag or constant of data type WORD (C#0 to C#999). |
| S1, S0, L1, L0, V1, V0, A1, R1 | CU | COUNTER | Count up:   As soon as the defined event occurs, the counter is incremented by "1". The counter value can be incremented until the limit of "999" is reached. When the limit is reached, the counter value is no longer incremented on a positive edge. |
| S1, S0, L1, L0, V1, V0, A1, R1 | CD | COUNTER | Count down:   As soon as the defined event occurs, the counter is decremented by "1". The counter value can be decremented until the limit of "0" is reached. When the limit is reached, the counter value is no longer decremented on a positive edge. |
| S1, S0, L1, L0, V1, V0, A1, R1 | CR | COUNTER | Reset counter:   As soon as the defined event occurs, the counter is reset to "0". |

### Permanent instructions (S7-300, S7-400, S7-1500)

#### Introduction

Permanent instructions allow you to create program code that is executed before or after the sequencer is processed. This means you can program conditions and block calls that are processed in every cycle independently of the sequencer.

You can create permanent instructions with the LAD and FBD programming languages. A maximum of 32 LAD or FBD instructions can be programmed in one permanent instruction. You can use up to 250 permanent pre-instructions and 250 permanent post-instructions within a GRAPH function block. You transfer the result of logic operation of the conditions to an assignment for which the "Set" and "Reset" memory functions are also available. The operand used (e.g., a bit memory) can be queried in transition, interlock, supervision, and other permanent networks.

### Conditions (S7-300, S7-400, S7-1500)

#### Introduction

Conditions are binary states of the process that can be linked together as LAD or FBD elements (e.g., normally open contact, normally closed contact, AND logic operation, OR logic operation, comparator) according to the Boolean logic in the ladder/function block diagram. The result of logic operation (RLO) can influence individual actions of a step, the whole step, the advancement to the next step, or the entire sequencer.

You use conditions to program the following components of your GRAPH program:

- Permanent instructions
- Interlocks
- Supervisions
- Transitions

Depending on the network, you can program up to 32 instructions to which a maximum of 32 operands are interconnected. You can have additional information about the utilized tags displayed by showing the tag information.

#### Basics on the block interface of a GRAPH function block (S7-300, S7-400, S7-1500)

##### Interface parameter sets

In the settings "Options > Settings > PLC programming > GRAPH" you can choose between the following interface parameter sets, each of which provides you with a different set of input and output parameters:

- Minimum interface parameter set

  The minimum interface parameter set contains only the input parameter "INIT_SQ" and no output parameters.
- Standard interface parameter set

  The standard interface parameter set enables operation of the sequencer in various operating modes and provides the option of acknowledging alarms.
- Maximum interface parameter set

  The maximum interface parameter set provides additional diagnostics.

You can manually delete or insert individual parameters in all parameter sets.

> **Note**
>
> When selecting, also take into consideration the available memory space in your CPU, because the memory space required for the GRAPH function block and the associated instance data block increases as the number of parameters increases.

See also:

[Input parameters](#input-parameters-s7-300-s7-400-s7-1500)

[Output parameters](#output-parameters-s7-300-s7-400-s7-1500)

##### Memory space models for the instance data block (S7-300/400)

For CPUs of the S7-300/400 series, you can additionally specify the memory space model of the instance data block. You have the following options:

- Standard data block
- Minimized data block

You define the number of static parameters via the selection of the memory model. Generally, both memory models store information on the individual steps and transitions as static parameters. A separate structure with detailed information is created for each step and each transition. You receive more detailed information in a standard data block than in a minimized data block. However a standard data block requires considerably more memory space.

The following restrictions apply to a minimized data block:

- Some elements of the steps and transition structures are unavailable.
- The "Skip steps" option is unavailable.
- No alarms are available.
- Step and transition numbers must be consecutive. If not, the steps and transitions are automatically renumbered during compilation.
- No data is generated for the criteria analysis.

See also: [Static parameters](#static-parameters-s7-300-s7-400-s7-1500)

##### Access to interface parameters

Generally, with an S7-1500 CPU, you do not have write access to the GRAPH-internal interface parameters. However, the following components of the "RT_DATA > MOP" structure in the "Static" section that manage the operating modes allow write access:

- AUTO
- MAN
- TAP
- TOP
- LOCK
- SUP
- OFF
- INIT
- HOLD
- TMS_HALT
- OPS_ZERO
- S_PREV
- S_NEXT
- S_ON
- S_OFF
- T_PUSH
- ACK
- IL_PERM
- T_PERM
- ILP_MAN

These parameters react to a positive signal edge. For S7-300/400 CPUs, also avoid write access to parameters other than those listed. Otherwise, error-free processing of the sequencer cannot be guaranteed.

See also: [Other static parameters](#other-static-parameters-s7-300-s7-400-s7-1500)

##### Resetting registration requirements

When errors and faults are registered via the "REG_EF" input parameter, the following differences exist between the CPU families with respect to resetting the registration requirement:

- S7-300/400

  The registration requirement is retained until an interlock or supervision error occurs in a step. Only then is the "R1" event triggered, the associated action executed, and the registration requirement reset.
- S7-1500

  The registration requirement is always reset, regardless of whether interlock or supervision errors occur in the current cycle.

##### Basic information on extending the standard behavior of GRAPH (S7-1500)

###### Introduction

GRAPH function blocks contain both visible and invisible internal parameters in the block sections in the "Static" section. Visible parameters are:

- Internal data area "RT_DATA"
- Transition structure "Trans1"
- Step structure "Step1"

Invisible parameters are:

- Description and status arrays

If required, as in STEP 7 Classic, you can also access the invisible parameters. This requires that the GRAPH block as of version 4.0 and higher is used. In addition to the GRAPH function block, an SCL function block is also required as an expansion. Accesses to the individual arrays within the extension block are via offsets, which are available in the "RT_DATA" structure.

##### Creating an extension function block and assigning it to a GRAPH FB (S7-1500)

You can create an extension block and assign it to the GRAPH function block in order to access the GRAPH-internal parameters. The extension block can be used as a single instance or multi-instance.

###### Requirement

The current project is a V14 SP1 project.

###### Procedure

To create an extension block and assign it to a GRAPH FB, proceed as follows:

1. Create a new function block (FB) in the "GRAPH" programming language.
2. Open the block properties of the new GRAPH FB.
3. Under "General > Version" set the version of the GRAPH FB to >= V4.0.
4. Create a new function block (FB) in the "SCL" programming language. You can assign any name to this FB. This FB serves as an extension block.
5. Enter the following in/out parameters in the extension block in the block interface in the "InOut" section:

   - "io_RT_Data", data type "G7_RTDataPlus_V4" (e.g. for the block version V4.0)
   - "io_G7T", data type "Array[*] of G7_TransitionPlus_V4" (e.g. for the block version V4.0)
   - "io_G7S", data type "Array[*] of G7_StepPlus_V4" (e.g. for the block version V4.0)
   - "io_G7Arrays", data type "Array[*] of USInt"
6. Open the block properties of the GRAPH FB.
7. Enter the name of the SCL function block under "Attributes > Name of extension block".
8. Instantiate the extension block either as a single instance or multi-instance:

   - Single instance:

     Create an instance data block of the type extension block. Enter a parameter with extension block as the data type in the GRAPH FB in the block interface in the "InOut" section.
   - Multi-instance:

     Enter a parameter with extension block as the data type in the GRAPH FB in the block interface in the "Static" section.
9. Call the GRAPH FB in any other block, the instance data block for the GRAPH FB is automatically created.
10. If you have instantiated the extension block as a single instance, provide the corresponding parameter of the block call, which you defined in the "InOut" section of the GRAPH FB, with the instance data block of the extension block.

**Note**

**Note on project compatibility**

If your current project is not yet a V14 SP1 project, a corresponding message will be displayed. In order to be able to access the GRAPH-internal parameters via the extension block, your project must be upgraded to a V14 SP1 project. After upgrading, however, you are no longer able to work with a TIA Portal version lower than V14 SP1.

**Note**

If you want to use an extension block for several GRAPH FBs, enter the extension block in the attributes of the corresponding GRAPH FBs. However, create a separate instance data block of the extension block for each GRAPH FB. To define different behavior for the various GRAPH FBs, you can use the "Static" section in the block interface of the extension block.

###### Result

The extension block is prepared and you can access the GRAPH-internal parameters within it. The individual description and status arrays are contained in the collective array "io_G7Arrays". Access to them is via the offsets. An access of this type could appear, for example, as follows:

#t_NumberOfActiveSteps := #io_G7Arrays[#io_RT_DATA.OFFSETS.SAX_OFFSET] - 1;

### Modifying settings for internal parameters (S7-300, S7-400, S7-1500)

You can declare the internal parameters of a GRAPH block as visible and accessible from HMI. For S7-1500 CPUs, you can also declare the internal parameters as retentive. If you declare the internal parameters as retentive, their values are retained even after an operating state change from "STOP" to "RUN", for example following a power failure. As a result, the sequencer remains in the current state and continues running from the point at which it was interrupted after the change to "RUN".

Parameters that you have declared yourself in the interface of a GRAPH block are not affected by these settings.

#### Declaring internal parameters as visible and accessible from HMI

Proceed as follows to declare all internal parameters of a GRAPH block as visible and accessible from HMI:

1. Open the GRAPH block for which you want to change the setting.
2. In the "Edit" menu, select the "Internal parameters visible/accessible from HMI" check box. Clear this check box if you want to reset the setting.

#### Declaring internal parameters as retentive (S7-1500)

Proceed as follows to declare all internal parameters of a GRAPH block as retentive:

1. Open the GRAPH block for which you want to change the setting.
2. In the "Edit" menu, select the "Retentive internal parameters" check box. Clear this check box if you want to reset the setting.

> **Note**
>
> **Points to note for S7-1500 CPUs with a firmware version lower than V1.5**
>
> When you declare the internal parameters of a GRAPH block as retentive, the sequence control remains in the current status and continues from the point at which it was interrupted after a change to "RUN". Please note that supervisions should not be used if you activate the new parameter "Retentive internal parameters" in a GRAPH block.

#### Set data in standard range as retentive (S7-300/400)

To set the data within the standard range as retentive, follow these steps:

1. Open the "Program blocks" folder in the project navigator.
2. Right-click on the instance data block of the GRAPH block.
3. Select the "Properties" command in the shortcut menu.

   The properties dialog box of the block opens.
4. Click "Attributes" in the area navigation.
5. Select the check box "Set data in standard range as retentive".
6. Confirm your entries with "OK".

### Inserting the sequencers (S7-300, S7-400, S7-1500)

When you open a GRAPH function block, the "Sequences" pane is expanded by default in the navigation view. A new GRAPH function block already contains a sequencer with one step and one transition. You can insert additional sequencers at any time.

#### Requirement

A GRAPH function block is open.

#### Procedure

To insert a new sequencer, follow these steps:

1. In the navigation view, right-click the title bar of the "Sequences" pane or the title bar of a sequencer that is already present.
2. Select the "Insert sequence" command in the shortcut menu.

   The sequencer is inserted after the selected sequencer. If no sequencer is selected, it is inserted before the first sequencer.

### Ending sequencers (S7-300, S7-400, S7-1500)

Every sequencer must be ended with a jump to another step in the same sequencer or another sequencer or with the "Sequence end" element. A jump to another step enables cyclic processing of the sequencer.

#### Requirement

A sequencer is open in the sequence view.

#### Procedure

To end a sequence, follow these steps:

1. Right-click the double arrow at the end of the sequencer.
2. Select the "Sequence end" element in the "Insert element" submenu of the shortcut menu.

Or:

1. Right-click the double arrow at the end of the sequencer.
2. Select the "Jump" element in the "Insert element" submenu of the shortcut menu.

   An overview containing all available steps of the GRAPH function block opens.
3. Select a step as the jump destination.

#### Insert steps and transitions (S7-300, S7-400, S7-1500)

When you insert a new GRAPH function block in your project, it already contains a sequencer with one step and one transition. You can insert additional steps and transitions into the sequencer.

Note the following rules when inserting steps and transitions:

- Steps and transitions must always alternate. This means that you cannot insert two steps or two transitions in a row.
- It follows from this that you can insert steps and transitions only at the end of a branch or at positions at which a step or transition is missing.
- A simultaneous branch must always start with a step.
- An alternative branch must always start with a transition.
- You can insert a step and a transition together as a pair. Because this method ensures that the step is followed by a transition, it is possible to do this at any position in the sequencer.

##### Requirement

A GRAPH function block is open in the sequence view.

##### Inserting a step

To insert a step, follow these steps:

1. Right-click a position in the sequencer where it is permitted to insert a step.
2. Select the "Insert element > Step" command from the shortcut menu.

Or:

1. Drag the "Step" element from Favorites or the "Instructions" task card to the appropriate position in the sequencer.

##### Inserting a transition

To insert a transition, follow these steps:

1. Right-click a position in the sequencer where it is permitted to insert a transition.
2. Select the "Insert element > Transition" command from the shortcut menu.

Or:

1. Drag the "Transition" element from Favorites or the "Instructions" task card to the appropriate position in the sequencer.

##### Inserting a step and transition together as a pair

To insert a step and a transition together as a pair, follow these steps:

1. Right-click the position in the sequencer where you want to insert the step and the transition.
2. Select the "Insert element > Step and transition" command from the shortcut menu.

Or:

1. Drag the "Step and transition" element from Favorites or the "Instructions" task card to the appropriate position in the sequencer.

#### Specifying and canceling the initial step (S7-300, S7-400, S7-1500)

When you insert a new GRAPH function block in your project, it already contains a sequencer with one step and one transition. By default, this step is specified as the initial step. This means that the sequencer is processed starting with this step. You can define a different step as the initial step at any time. In simultaneous branches, it is also permitted to have multiple initial steps. However, the maximum number of initial steps is 250. You can change an initial step to a normal step subsequently by canceling it.

##### Requirement

A GRAPH function block is open in the sequence view.

##### Specifying an initial step

To specify a step as an initial step, follow these steps:

1. Right-click the step you want to specify as an initial step.
2. Select the "Initial step" check box in the shortcut menu.

##### Canceling an initial step

To cancel an initial step and use it as a normal step again, follow these steps:

1. Right-click the step you want to specify as an initial step.
2. Clear the "Initial step" check box in the shortcut menu.

#### Renumbering steps and transitions (S7-300, S7-400, S7-1500)

When you insert steps and transitions in your sequencer, the next available number is assigned. You can change this number subsequently. You have the following two options for this:

- You manually assign new numbers for the steps and transitions you want to renumber. Note that the same number must not be assigned more than once.
- You perform program-supported renumbering This method allows you to have all steps and transitions of a branch, sequencer, or block to be renumbered automatically. Observe the following differences when using this method:

  - Renumbering within a branch:

    Within the branch, all steps or transitions starting from the selected step or transition are renumbered. The renumbering ends at the start of a new branch.
  - Renumbering within a sequencer:

    Within the sequencer, all steps or transitions are renumbered. The first step or the first transition of the sequencer is assigned the start number you entered. Note that steps in other sequencers may also be renumbered in the process if duplicate numbers would otherwise result.
  - Renumbering within a block:

    Within the GRAPH block, all steps or transitions are renumbered. The first step or the first transition within the first sequencer of the block is assigned the start number you entered.

  You can also renumber only the steps or only the transitions.

  > **Note**
  >
  > Note, however, that this option is not available for GRAPH function blocks for which you have selected the "Create minimized DB" attribute.

##### Requirement

A GRAPH function block is open in the sequence view.

##### Manual renumbering of steps and transitions

To manually renumber steps or transitions, follow these steps:

1. Select the step whose number you want to change.
2. Click the number of the step.
3. Enter the new number.

##### Program-supported renumbering of steps and transitions within a branch

For program-supported renumbering of steps or transitions within a branch, follow these steps:

1. Right-click the step or transition where you want to start renumbering the branch.
2. Select the "Renumber" command in the shortcut menu.

   The "Renumber" dialog opens.
3. If you want to renumber the steps, select the "Steps" check box and enter the number to be assigned to the selected step.
4. If you want to renumber the transitions, select the "Transitions" check box and enter the number to be assigned to the transition following the selected step.
5. Select the "Branch" option button in the "Renumber in" area.
6. Click "OK" to confirm.

##### Program-supported renumbering of steps and transitions within a sequencer

For program-supported renumbering of steps or transitions within a sequencer, follow these steps:

1. Right-click a step or transition within the sequencer for which you want to carry out renumbering.
2. Select the "Renumber" command in the shortcut menu.

   The "Renumber" dialog opens.
3. If you want to renumber the steps, select the "Steps" check box and enter the number to be assigned to the first step of the sequencer.
4. If you want to renumber the transitions, select the "Transitions" check box and enter the number to be assigned to the first transition of the sequencer.
5. Select the "Sequence" option button in the "Renumber in" area.
6. Click "OK" to confirm.

##### Program-supported renumbering of steps and transitions within a GRAPH function block

For program-supported renumbering of steps or transitions within a GRAPH function block, follow these steps:

1. Right-click a step or transition within the GRAPH function block for which you want to carry out renumbering.
2. Select the "Renumber" command in the shortcut menu.

   The "Renumber" dialog opens.
3. If you want to renumber the steps, select the "Steps" check box and enter the number to be assigned to the first step of the GRAPH function block.
4. If you want to renumber the transitions, select the "Transitions" check box and enter the number to be assigned to the first transition of the GRAPH function block.
5. Select the "Block" option button in the "Renumber in" area.
6. Click "OK" to confirm.

#### Renaming steps and transitions (S7-300, S7-400, S7-1500)

When you insert steps and transitions into your sequencer, they are initially named "Step<StepNumber>" and "Trans<StepNumber>". You can change these names in order to make your program easier to understand. Note that the same name must not be assigned more than once.

##### Requirement

A GRAPH function block is open in the sequence view.

##### Renaming a step

To rename a step, follow these steps:

1. Select the step whose name you want to change.
2. Click the step name.
3. Enter a new name.

##### Renaming a transition

To rename a transition, follow these steps:

1. Double-click the transition name you want to change.
2. Enter a new name.

> **Note**
>
> **Changing names in the "Texts" tab**
>
> If you change the name of the step or transition in the "Texts" tab, the changes are not displayed in the programming editor. The step and transition names displayed in the programming editor cannot be translated. They correspond to the names of the corresponding block interface elements in the "Static" section of the GRAPH block and the names of block interface elements cannot be translated.

The translations of the step and transition names are displayed both in the message texts and in the viewers. The default step and transition name is retained in the programming editor.

   If you want to display the translations of all steps and transitions simultaneously, click in the GRAPH FB outside the sequencer.
2. Open the "Properties" tab in the Inspector window.
3. Open the "Texts" tab.

   If the names of the steps and transitions appear only in the current user interface language, either you have not yet selected any project languages or translations are not yet available. You can enter the translations directly in the table on the "Texts" tab or you can enter the translations for the individual project languages in the global "Project texts" table. You can find additional information on translation of texts under [Project text basics](Editing%20project%20data.md#project-text-basics).​

### Inserting actions (S7-300, S7-400, S7-1500)

#### Introduction

You can specify actions that will be implemented when the step is processed. In order to add comments for actions or adapt block calls, you can enable a multi-line input. Single-line input is the default setting. You can insert actions either in the single step view or the sequence view.

For a better overview of your programmed actions you can show descriptions for the events and identifiers.

In actions you can assign an operand with the value of another operand or mathematical expression or the result of an instruction. Use the following syntax for this:

- Assigning the value of an operand: A:=B
- Assigning the value of a mathematical expression: A:=B<Operator>C

  You can use the following operators: + (Addition), - (Subtraction), * (Multiplication), / (Division), % (Modulo), AND (AND logic operation), OR (OR logic operation), XOR (Exclusive OR logic operation)
- Assigning the result of an instruction: A:=<Instruction>(B)

  You obtain the available instructions from the "Instructions" task card.

Note that the data types of the operands must be compatible with one another.

#### Inserting actions in the single step view

To insert an action in the single step view, follow these steps:

1. Open the single step view.
2. If the actions are collapsed, click the small arrow in front of "Actions".

   The Right arrow becomes a Down arrow, and the table containing the actions is displayed.
3. If you want to enable multi-line input, right-click in the line of the new action and select the "Allow multi-line mode" command in the shortcut menu.
4. If you want to link the new action with an interlock, click in the cell of the "Interlock" column and select the "-(C)- - Interlock" entry in the drop-down list.
5. If you want to link the new action with an event, click in the cell of the "Event" column and select the appropriate event in the drop-down list.
6. Click in the cell of the "Identifier" column and select the identifier of the new action in the drop-down list.
7. In the "Action" column, specify the action that is to be performed. You have the following options:

   - You can replace the placeholders with the operands or values that you want to use for the action. You can also insert these using drag-and-drop operation or via autocompletion.
   - You can use instructions from the "Instructions" task card. To do so, drag-and-drop them from the task card into the "Action" column.
   - You can drag-and-drop blocks from the project tree into the "Action" column in order to call them.

**Note**

When data-type-dependent instructions are called, you must specify a data type. You can either manually enter the data type or select it from a drop-down list by clicking the small green rectangle that is shown below the existing data type.

#### Inserting actions in the sequence view

To insert an action in the sequence view, follow these steps:

1. Open the sequence view.
2. Expand the step in which you want to insert actions. To do so, click the button at the top right corner of the step or select the step and press <Enter>.

##### Basics of step time monitoring (S7-1500)

###### Introduction

With supervision conditions, you can monitor the step activation time; in other words, the time that has elapsed since activation of the step. To do this, you can specify the maximum time "T_MAX" and use this as a comparison value. To make the comparison, you can use the instruction "Greater than maximum step activation time" (CMP>T_MAX). If the maximum step activation time is exceeded, a supervision error is triggered and a supervision alarm is output, if you have enabled this.

To receive a warning before the supervision error occurs, you can specify a warning time "T_WARN" and monitor this, once again using a condition. To do this, you can use the instruction "Greater than warning time" (CMP>T_WARN) and a warning coil. If the warning coil is activated, an alarm is output. Note that the alarm is only present in the current cycle, even if the warning coil remains in the "1" state for longer. Therefore, to display the alarm, use the alarm archive or assign the alarm to an alarm class requiring acknowledgment so that the alarm is not moved to the alarm archive until it has been acknowledged.

Initially, the values specified in the settings in "PLC programming > GRAPH > Default for time monitoring" are used for "T_MAX" and "T_WARN". You have the following options for changing the times for "T_MAX" and "T_WARN":

- [Manually using the "Define time" dialog](#specifying-times-t_max-and-t_warn-manually)
- [Using the learning mode of the interface](#determining-times-for-t_max-and-t_warn-using-the-learning-mode-of-the-interface-s7-1500)
- [Using the learning mode when testing](Testing%20GRAPH%20programs%20%28S7-300%2C%20S7-400%2C%20S7-1500%29.md#calculating-times-for-t_max-and-t_warn-when-testing-s7-1500)

If you use the learning mode, the activation time of the steps is measured as the program runs. The longer you leave the learning mode activated, the more accurately the activation time can be determined. The times for "T_MAX" and "T_WARN" result from the sums of the measured times and the additive thresholds that you can set.

If you use the learning mode to determine the times during testing, you can, if necessary, reset the learned times. This returns "T_MAX" and "T_WARN" to their default values in the instance data block.

> **Note**
>
> For GRAPH blocks with a version lower than 2.0, the learning mode for determining the values of "T_MAX" and "T_WARN" is not available. You can change the version in the properties of the GRAPH block under "General > Block".

##### Determining times for T_MAX and T_WARN using the learning mode of the interface (S7-1500)

You can have the times for "T_MAX" and "T_WARN" determined directly using the block call of the GRAPH function block. However, note the following restrictions as compared to the use of the learning mode when testing that you can start in the "Testing" task card:

- The determined values are only saved in the online instance data block.
- You cannot reset the learned times.

To make sure that "T_MAX" and "T_WARN" are greater than the learned activation times, you can define additive thresholds that are added to the learned times. Enter a lower value for the warning than for the supervision so that the warning is displayed before the supervision.

###### Requirement

The GRAPH function block is called in the program.

###### Defining additive thresholds

To define additive thresholds, follow these steps:

1. Insert the "MOVE" instruction in the block that calls the GRAPH function block.
2. For the "IN" parameter, enter the additive threshold for the supervision as a percentage.
3. For the "OUT1" parameter, enter the static parameter "RT_DATA > TRESHOLD_SUP" of the instance data block.
4. Insert another "MOVE" instruction.
5. For the "IN" parameter, enter the additive threshold for the warning as a percentage.
6. For the "OUT1" parameter, enter the static parameter "RT_DATA > TRESHOLD_WARN" of the instance data block.

   The additive thresholds are taken into account in the calculation of "T_MAX" and "T_WARN" in learning mode.

###### Enabling learning mode

To activate learning mode via the interface, follow these steps:

1. Set the interface parameter set to "Maximum interface parameters" or insert the two parameters manually in the block interface of the GRAPH function block:

   - "EN_LMODE" input parameter
   - "LMODE_ENABLED" output parameter
2. Open the calling block.
3. Link the "EN_LMODE" input parameter to a tag.
4. Download the block to the device.
5. In the toolbar, click "Monitoring on/off" to start testing with program status for the calling block.
6. Right click on the tag you linked to the "EN_LMODE" input parameter.
7. In the shortcut menu, select the command "Modify > Modify to 1" to obtain a rising edge for the "EN_LMODE" input parameter.

   The signal state of "LMODE_ENABLED" is set to "1" and the learning mode is activated.
8. Wait until each step in the GRAPH function block has been active at least once.
9. Right click again on the tag you linked to the "EN_LMODE" input parameter.
10. In the shortcut menu, select the command "Modify > Modify to 0" to reset the "EN_LMODE" input parameter.
11. Right click again on the tag you linked to the "EN_LMODE" input parameter.
12. In the shortcut menu, select the command "Modify > Modify to 1" to obtain a rising edge for the "EN_LMODE" input parameter.

    Learning mode is deactivated. The learned times are entered in the online instance data block of the GRAPH function block.

##### Inserting network title (S7-300, S7-400, S7-1500)

The network title is the header of a network. The length of the network title is limited to one line. You can enter the title manually or set it automatically. For automatic insertion of the network title, the comment of the operand in one of the following instructions in the network is evaluated:

- Assignment
- Set output
- Reset output

- The network does not have a title yet.
- The operand of the instruction used for the comment has a comment.

> - The network title is only set by the writing instructions listed above.
> - If the operand is of the data type array, the comment of the array is used and not the comments of the array elements.
> - Comments of invalid operands are not taken into consideration.

### Displaying or hiding tag information (S7-300, S7-400, S7-1500)

#### Introduction

For permanent instructions, interlocks, supervisions, and transitions, you can display the following information on the utilized tags in the programming window:

- Name of the tag
- Tag addresses
- Simple or hierarchical comments to document the tags

The information is taken from the block interface for local tags and DB tags and from the PLC tag table for tags that are valid CPU-wide.

You can display the tag information either for all the blocks or for individually opened blocks. If you display the tag information for all the blocks, the tag information for all the blocks currently opened and opened in future is shown. You can hide the tag information at any time again. If you have hidden the tag information for all blocks, you can display it again for individual blocks that are open.

When you select the display of tag information with hierarchical comments, the comments of the higher structure levels are also displayed for structured tags. The comments are shown in brackets after the tag comment; the comments of the individual levels are separated by a dot. If there is no comment for a tag on a structure level, it is not displayed. This is indicated by two successive dots.

You can display the tag information either collectively below a LAD/FBD network or directly at the respective operand. As with showing and hiding tag information, you can set this via the settings for all block or change specifically for an opened block.

#### Displaying or hiding tag information for all the blocks

Proceed as follows to display or hide the tag information for all the blocks:

1. Select the "Settings" command in the "Options" menu.

   The "Settings" window is displayed in the work area.
2. In the area navigation, select the "PLC programming" group.
3. If you want to display the tag information, select the "Show" option or the "Tag information with hierarchical comments" option in the drop-down list, depending on whether you want to display simple or hierarchical comments.
4. Use the "Position of the tag information (LAD/FBD)" drop-down list to specify whether the tag information is to be displayed below the LAD/FBD network or on the operand.

   For all open blocks, the tag information is shown and displayed at the selected position. If you open additional blocks the tag information for these blocks is also shown.

### Updating block calls in GRAPH (S7-300, S7-400, S7-1500)

If interface parameters of a called block are changed, the block call can no longer be executed correctly. You can avoid such inconsistent block calls by updating the block calls.

You have the following options for updating the block calls:

- Explicit updating of all inconsistent block calls in the programming editor.

  The inconsistent block calls within the open block are updated. The following actions are carried out in the process:

  - New parameters are added. Please note, however, that the parameters are not shown for function blocks (FBs) and the parameters are supplied via the corresponding instance data block (DB).
  - Deleted parameters are not removed.
  - Renamed parameters get the new parameter names.
  > **Note**
  >
  > If updating all inconsistent block calls would cause an error in the parameter supply, you cannot use the "Update block calls" command. If this is the case, update each block call individually.
- Explicit updating of a block call in the programming editor.

  The inconsistent call of this block is then updated. The following actions are carried out in the process:

  - New parameters are added. Please note, however, that the parameters are not shown for function blocks (FBs) and the parameters are supplied via the corresponding instance data block (DB).
  - Deleted parameters are not removed.
  - Renamed parameters get the new parameter names.
- Implicit updating during compilation.

  All block calls in the program as well as the used PLC data types will be updated. Note that when you call functions (FCs) before the compilation process, all new formal parameters must be supplied with actual parameters.

#### Updating all inconsistent block calls in the programming editor

To update all block calls in a block, follow these steps:

1. Open the calling block in the programming editor.
2. Click the "Update inconsistent block calls" button in the toolbar.

#### Updating a specific block call in the programming editor

To update a specific block call in the programming editor, follow these steps:

1. Open the calling block in the programming editor.
2. Right-click on the block call that you want to update.
3. Select the "Update block call" command in the shortcut menu.
4. If parameters were added for a function (FC), enter the values for the new block parameters.

> **Note**
>
> Note that the "Update" command is only available as long as you did not previously update all block calls in the editor with the "Update inconsistent block calls" command.

#### Updating block calls during compilation

To implicitly update all block calls and uses of PLC data types during compilation, follow these steps:

1. Open the project tree.
2. Select the "Program blocks" folder.
3. Select the "Compile > Software (rebuild all blocks)" command in the shortcut menu.

### Overview of initial value acquisition (S7-1500)

#### Requirement

In order to enable initial value acquisition, you need to set a version >= 4.0 in the GRAPH function block.

You can find additional information here: [Activating initial value acquisition](#activating-initial-value-acquisition-s7-1500)

#### Description

You have the option of testing the execution of your user program on the HMI device. The data and values on the HMI device are continuously synchronized with the CPU and updated. You can always see the current program status on the HMI device for each GRAPH function block. The actual values of the signal states of Boolean operands and the results of comparator operations in interlocks and transitions are recorded continuously from the time of activation of the initial value acquisition, even if no error occurs.

The signal states are stored for each cycle in the GRAPH instance data block. A maximum of 32 signal states can be recorded per interlock or per transition. The actual values are stored in the static parameters CRIT_LOC (step) or CRIT (transition) in a DWORD and each signal state occupies one bit. As soon as an interlock error occurs, the values of the CRIT_LOC parameter are copied to the CRIT_LOC_ERR parameter. As soon as a supervision error occurs, the values of the CRIT parameter are copied to the CRIT_FLT parameter in the next transition. These initial values are no longer changed when a new error occurs in the active step.

You can display the actual values with the help of the PLC code view. As soon as an error occurs, you can have the initial values displayed and the bad operands can be immediately recognized (criteria analysis). You can switch between display of the actual values and display of the initial values.

You can find additional information on the visualization of the HMI device here: "Visualize processes > Using diagnostics functions > Supervising machinery and plants with ProDiag"

For the named static parameters of the step or transition structures to be displayed, you need to select the maximum interface parameter set for the block interface.

You can find additional information on the interface parameters here: [Specifying the interface parameter set](#specifying-the-interface-parameter-set-s7-300-s7-400-s7-1500)

You can find additional information on the static parameters of a GRAPH function block here: [Static parameters](#static-parameters-s7-300-s7-400-s7-1500)

#### Supported instructions

All instructions supported in a GRAPH function block for transitions and interlocks are supported for the initial value acquisition.

For bit logic operations, the status of the operand is stored. For comparators, the result of the comparison is stored.

#### Recording actual values or initial values

The respective interlock of the active steps and subsequent transitions is recorded in each program cycle. If you enable the block property "Permanent processing of all interlocks in manual mode" or the test settings "Edit all interlocks" or "Edit all transitions", all interlocks and all transitions of the GRAPH sequencer will be recorded in every program cycle. That may result in a longer cycle time.

#### Evaluation order on the HMI device in the PLC code view

The recording of the signal status in the respective DWORD or the list of operands in the symbol table of the PLC code view is carried out in a defined order. In a LAD interlock, for example, the process always starts in the power rail at the top left and ends at the bottom right.

> **Note**
>
> **Observe the evaluation order during programming**
>
> Also in the criteria analysis on the HMI device, this order is adhered to for the list of the bad operands in the symbol table. Observe this evaluation order for the programming of the interlock or the transition.

Additional information on the order of the list is available here: [Example of order of the list](#example-of-order-of-the-list-s7-1500)

### Activating initial value acquisition (S7-1500)

#### Procedure

Proceed as follows to activate initial value acquisition:

1. Right-click the GRAPH function block and select "Properties..." from the shortcut menu.

   The properties dialog of the GRAPH function block opens.
2. Select a version >= V4.0 in the "General > Block" tab.
3. Click on the "Attributes" tab.
4. Activate "Initial value acquisition".
5. Click "OK".
6. Compile and download the user program.

#### Result

The recording of the initial values begins once the user program has been compiled.

### Resetting initial values (S7-1500)

#### Description

Each GRAPH function block contains the two system-internal parameters "H_IL_ERR" and "H_SV_FLT" in its block parameter set for each individual step. If you have activated the "Initial value acquisition" option in the block properties of the GRAPH function block, the Bit 2 at the "H_IL_ERR" (Interlock) parameter is set to TRUE in the case of an interlock, and the Bit 5 at the "H_SV_FLT" (Transition) parameter is set to TRUE in the case of a monitoring error, as soon as a corresponding event exists in the active step. This enables you to visualize the initial values on your HMI device with the PLC code view.

As of a block version >= 5.0 of a GRAPH function block, you have the option of using input parameter "RESET_CRIT" to reset the signal state of the two static parameters "H_IL_ERR" and "H_SV_FLT" to FALSE again after the criteria analysis. You should use this option, in particular, when you change the programming of interlocks and transitions and you no longer need the initial values that were made obsolete by this. The reprogramming does not automatically reset the signal states back to FALSE.

#### Procedure

To reset the signal states of the two system-internal static parameters, follow these steps:

1. Check that the version of the GRAPH function block is a version >= 5.0.
2. Select the maximum block parameter record:

   - When you want to create a new GRAPH function block, first select the maximum block parameter record in the general settings for GRAPH.
   - If you want to change the parameter record at a GRAPH function block that has already been created, open the block, click in the block interface and in the menu bar select "Edit > Interface parameters > Maximum interface parameters". Or create the RESET_CRIT parameter manually in the IN section.
3. Connect input parameter "RESET_CRIT" in your user program at the call of the GRAPH function block.

#### Result

As soon as a positive signal edge is present at input parameter "RESET_CRIT", the signal states of the two system-internal static parameters "H_IL_ERR" and "H_SV_FLT" are reset to FALSE in all the programmed GRAPH steps and the PCL code view on your HMI device no longer displays any obsolete events.
