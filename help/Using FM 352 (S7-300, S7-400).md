#### Auxiliary voltage for encoders and digital outputs (L+, M)

The 24 VDC auxiliary voltage of the encoders and digital outputs is monitored:

- for wire-break of the 24 V feed line
- for power failure

This supplies 24 V DC and 5.2 V DC at the encoder interface (sub D socket X2) to the various types of encoders.

#### Load current supplies

The DC load power supply must meet the following requirements:

Only a safe, isolated extra-low voltage of ≤ 60 V DC may be used as the load current supply. Safe isolation can be implemented, for example, by adhering to the specifications

- VDE 0100 Part 410 / HD 384-4-41 / IEC 364-4-41  
   (as protective extra low voltage) and/or
- VDE 0805 / EN 60950 / IEC 950  
  (as safety low voltage SELV) or
- VDE 0106 Part 101.

#### 4 digital inputs (I0 to I3)

You can connect bounce-free switches (24 V current sourcing) or non-contact sensors (2- or 3-wire proximity switches) to the 4 digital inputs.

The digital inputs are not monitored for short circuits or wire break and have a non-isolated connection to the module chassis.

#### 13 digital outputs (Q0 to Q12)

The state (on/off) of tracks 0 to 12 is output at 13 digital outputs. The digital outputs have a non-isolated connection to the module chassis.

Loads supported:

- Operating voltage 24 V
- Current load 0.5 A / short-circuit proof

The state of each output can be read off from the respective LED.

### Influence of functions on the switching characteristics of time-based cams (S7-300, S7-400)

#### Actual value changes

A time-based cam can be skipped by the following functions that change the actual value:

- Set actual value
- Set actual value on-the-fly
- Zero offset
- Retrigger reference point

#### Activating a time-based cam

If you skip the start of a time-based cam with one of the above functions, this cam will be activated provided the actual direction of the axis motion is the same as the effective direction assigned for the cam. The programmed cam activation time starts.

> **Note**
>
> If the axis is at a standstill, the direction of movement is influenced by fluctuations of the actual value.
>
> Set a hysteresis higher than the fluctuation of the actual value signal in order to suppress flutter when the axis is at a standstill.
>
> This retains the last determined direction of movement while the axis is at a standstill.

| Symbol | Meaning |
| --- | --- |
|  | **Warning** |
| Risk of injury and material damage.  Functions which modify the actual value at rotary axes may cause the unwanted activation of time-based cams.  You should always set the "invalid" option for the time-based cams of a rotary axis if you want to influence the actual value using the functions mentioned above. |  |

### Set Actual Value/Set Actual Value on-the-fly/Cancel Set Actual Value (S7-300, S7-400)

#### Definition

Use the "Set process value" and "Set process value on-the-fly" functions to assign a new coordinate to the current encoder value. This shifts the coordinate system by the value: ACT<sub>new</sub> - ACT<sub>current</sub>

Whereby:

- ACT<sub>new</sub> is the default value
- ACT<sub>current</sub> is the process value at the time of execution

#### Determining new coordinates

Calculate all your default positions in the shifted coordinate system based on the following formula:

**Coordinate**
<sub>new</sub>
 **= Coordinate**
<sub>old</sub>
 **+ (ACT**
<sub>new</sub>
 **- ACT**
<sub>current</sub>
**)**

#### Requirements

- The axis must be synchronized.
- With "set process value on-the-fly": Digital input I1 must be interconnected.

#### Function sequence

1. Enter the coordinate for the process value, or for the process value to set on-the-fly in the channel DB.

   - Linear axis:

     You must select the specified process value so that the software limit switches are still within the valid traversing range after the function is called.

     The absolute value of the offset derived from **(ACT**<sub>new</sub>**- ACT**<sub>current</sub>**)** must be less than or equal to the absolute value of the permitted traversing range (maximum 100 m or 1000 m).
   - Rotary axis:

     Rule for the defined process value:

     0 ≤ process value < end of rotary axis
2. Set the appropriate trigger bits at the channel DB.
3. Call the [CAM_CTRL](FM%20x52%20Cam%20Controlling%20%28S7-300%2C%20S7-400%29.md#cam_ctrl-s7-300-s7-400) instruction.

   "Set process values" is executed immediately.

   "Set process value on-the-fly" is executed at the next positive edge at digital input I1. The FVAL_DONE bit is set.

#### Data used in the channel DB

| Address | Name | Type | Initial value | Comment |
| --- | --- | --- | --- | --- |
| 36.4 | AVAL_EN | BOOL | FALSE | 1 = set process value |
| 36.5 | FVAL_EN | BOOL | FALSE | 1 = set process value on-the-fly |
| 90.0 | AVAL | DINT | L#0 | Process value coordinate |
| 94.0 | FVAL | DINT | L#0 | Coordinate for on-the-fly process value |
| 25.5 | FVAL_DONE | BOOL | FALSE | 1 = set process value on-the-fly executed |

#### Effects of the function

The example "set process value" to 400 mm (at position 200 mm) demonstrates how this function shifts the coordinate system. Resultant effects:

- The position of the working range is **not** physically shifted.
- The individual points (such as the software limit switches) are assigned new coordinate values.
- The cams retain their coordinate values, and are therefore located at a different physical position.
- If the axis is synchronized and cam processing is enabled, the positional process value may skip cam edges or entire cams as a result of this function.
- Status changes of the cam which would normally trigger an interrupt may be lost.

  > **Note**
  >
  > Make allowances for the [switching characteristics of time-based cams](#influence-of-functions-on-the-switching-characteristics-of-time-based-cams-s7-300-s7-400).

The table below shows how "Set process value" / "Set process value on-the-fly" shifts the coordinate system.

| Set process value | SLS  [mm] | REF  [mm] | ACT  [mm] | SLE  [mm] |
| --- | --- | --- | --- | --- |
| ![Effects of the function](images/24633497099_DV_resource.Stream@PNG-en-US.png) | -400 | -200 | 200 | 400 |
| -200 | 0 | 400 | 600 |  |

#### Cancelling the function

The "Undo set process value" function can be used to reset the coordinate shift caused by "Set process value" or "Set process value on-the-fly".

Once "Set process value on-the-fly" has been triggered, it can no longer be deleted before execution by a positive edge at input I1. However, it can be overwritten by a new "Set process value on-the-fly".

These functions are reset when a module is started up.

#### Data used in the channel DB

| Address | Name | Type | Initial value | Comment |
| --- | --- | --- | --- | --- |
| 35.2 | AVALREM_EN | BOOL | FALSE | 1: Cancel process value setting |

#### Possible causes of error

"Set process value on-the-fly" and "Retrigger reference point" may not be executed in parallel.

The "Set actual value on-the-fly" function can signal an operator error if the function would cause a software limit switch to exceed the valid traversing range with a positive edge at I1. This operating error is signaled by a diagnostic interrupt, and written to the diagnostic buffer.

### Set reference point  (S7-300, S7-400)

#### Definition

The "set reference point" function is used to synchronize the axis. The function shifts the working area. All shifts generated by a zero offset or "set actual value" are retained.

The function projects the operating range on the axis. Different value settings will thus move the working range to different positions within the physical range of the axis.

#### Requirements

Cam processing must be disabled.

#### Function sequence

1. Write the value for the reference point coordinate to the channel DB.

   - Linear axis:

     The reference point coordinates must not lie outside the software limit switches. This also applies to the reference point coordinate in a shifted coordinate system.
   - Rotary axis:

     Rule for the reference point coordinate:

     0 ≤ reference point coordinate < end of rotary axis
2. Set the relevant trigger bit.

#### Data used in the channel DB

| Address | Name | Type | Initial value | Comment |
| --- | --- | --- | --- | --- |
| 36.3 | REFPT_EN | BOOL | FALSE | 1 = set reference point coordinate |
| 98.0 | REFPT | DINT | L#0 | Reference point coordinate |
| 25.0 | SYNC | BOOL | FALSE | 1 = axis synchronized |

#### Effects of the function

The example "Set reference point" to 300 mm demonstrates how this function shifts the axis operating range.

This has the following effects:

- The actual position is set to the value of the reference point coordinate.
- The working range is physically shifted on the axis.
- The various points retain their original coordinates, but are now at new physical positions.
- The SYNC bit is set in the checkback signals.

### Change cam edges  (S7-300, S7-400)

#### Definition

The "change cam edges" function can be used to change the cam start and, for position-based cams, the end of a specific position-based cam in runtime.

#### Requirements

The cam you want to change must be valid.

#### Function sequence

1. Enter the cam number in the channel DB.
2. At a position-based cam:

   Enter the cam start and cam end in the channel DB.

   For a time-based cam:

   Enter the cam start value in the channel DB.
3. Set the relevant trigger bit.

#### Data used in the channel DB

| Address | Name | Type | Initial value | Comment |
| --- | --- | --- | --- | --- |
| 36.7 | CH01CAM_EN | BOOL | FALSE | 1 = write function for cam edges (1 cam) |
| 102.0 | CAM_NO | INT | 0 | Cam number |
| 104.0 | CAM_START | DINT | L#0 | Cam start |
| 108.0 | CAM_END | DINT | L#0 | Cam end |

#### Effects of the function

The FM x52 shifts first the on edge and then the off edge of the cam. This sequence does not depend on the direction in which the cam is shifted.

**Exception:**

The sequence described above may briefly generate an [inverse cam](Cam%20control%20%28S7-300%2C%20S7-400%29.md#properties-of-the-cam-types-s7-300-s7-400) if the new cam start is greater than the old cam end.

> **Note**
>
> If a hardware interrupt has been enabled for this cam, FM x52 can trigger one or two hardware interrupts when it detects the inverse cam depending on parameter assignment.

The modified values are lost when you restart the module.

### Perform "Fast Cam Parameter Change" (S7-300, S7-400)

#### Definition

The "Fast cam parameter change" function can be used to modify a group of up to 16 cams in runtime.

#### Requirements

2. Enter the number of the first cam to be modified in the channel DB.
3. Set the trigger bits for the required modifications.

#### Deactivating a cam when making modifications

To maintain consistency, always deactivate the cam (CAM_OFF) when modifying its start and end settings.

#### Data check by the module

Use the DIS_CHECK (channel DB) parameter to specify whether or not to disable the validation of transferred data by FM x52. If you disable the data check, you must take steps to verify that only permitted values are transferred. If you transfer illegal values without a check, the module can behave unexpectedly.

- FALSE: The module checks all data to be transferred
- TRUE: The check of cam parameter data is disabled. This allows for faster activation of the data to be changed on FM x52.

  Regardless of this function, the module always checks whether

  - the axis is assigned
  - the number of cams to be changed (C_QTY) is valid
  - the cam (cam number) to change is valid.

The data are only enabled on the module after having been checked and found faultless.

Any faulty data are rejected.

#### Effects of the function

> **Note**
>
> Make allowances for the [switching characteristics of time-based cams](#influence-of-functions-on-the-switching-characteristics-of-time-based-cams-s7-300-s7-400).

You can read the modified values by calling one of the jobs CAM1RD_EN to CAM8RD_EN.

#### Cancelling the function

The modified values are lost when you restart the module.

### Executing "Length measurement" and "Edge detection" (S7-300, S7-400)

#### Definition

You can establish the length of a workpiece using the "Length measurement" and "Edge detection" functions.

Length measurement and edge detection are active and remain active until you disable these functions or select a different measuring method. The [CAM_CTRL](FM%20x52%20Cam%20Controlling%20%28S7-300%2C%20S7-400%29.md#cam_ctrl-s7-300-s7-400) instruction enables length measurement if you select both measuring methods in parallel.

#### Requirement

A **bounce-free** switch must be connected to input I1.

#### Function sequence

The FM 352 updates the data on the module at different times depending on the type of measurement. FM 352 reports each update at a parameter on the feedback interface.

**Length measurement**

![Function sequence](images/20334001291_DV_resource.Stream@PNG-en-US.png)

1. Set the function switch for "length measurement".
2. A positive edge at input I1 starts the length measurement.
3. The negative edge at input I1 stops the current measurement. The FM 352 updates the initial value, final value and length data.
4. When parameter MSR_DONE is set, FM 352 reports the data update. The parameter indicates completion of the measurement. The results of the measurement can be read out.
5. The start of the next measurement with a rising edge at I1 resets the MSR_DONE parameter.

The FM 352 does not update the data if the function is disabled during length measurement. The MSR_DONE parameter remains reset.

**Edge detection**

![Function sequence](images/20336405259_DV_resource.Stream@PNG-en-US.png)

1. Enter a value for the minimum edge distance in the parameter DB. Write and enable machine data.
2. Set the "edge detection" function switch. This sets the MSR_DONE parameter.
3. The positive edge at input I1 starts edge detection. The results of the measurement are updated and can be read out, and the start value of the measurement is entered; the end value and length assume the value -1.
4. After the update, FM 352 reports the change by resetting the MSR_DONE parameter.
5. The negative edge at input I1 stops the current measurement. The FM 352 updates the data for the end measurement and length values.
6. After the update, FM 352 reports the change by setting the MSR_DONE parameter. The results of the measurement can be read out.
7. The start of the next measurement with a rising edge at I1 resets the MSR_DONE parameter.

The FM 352 does not update the data if the function is disabled during edge detection. The MSR_DONE parameter remains reset.

| --- | --- | --- | --- | --- |

| 34.0 | EDGE_ON | BOOL | FALSE | 1 = edge detection on |
| 34.2 | MSR_ON | BOOL | FALSE | 1 = length measurement on |
| 38.2 | MSRRD_EN | BOOL | FALSE | 1 = read measured values |
| 112.0 | BEG_VAL | DINT | L#0 | Initial value |
| 116.0 | END_VAL | DINT | L#0 | End value |

| Address | Name | Type | Initial value | Comment |
| --- | --- | --- | --- | --- |
| 4.0 | EDGEDIST | DINT | L#0 | **Minimum edge distance for edge detection**   Valid value range:   - 0 ... 1 000 000 000 µm |

The minimum edge distance is used to define a range after the start of a measurement is detected with an edge detection. The measurement will be discarded if the measuring operation ends within this range.

The start of the measurement is not signaled unless the minimum edge distance has been traveled.

#### Conditions of length measurement

- The CPU program requires an interval of sufficient length between the off and on edges at input I1 in order to be able to evaluate the result of the measurement before a new measurement is triggered.
- The minimum interval between the positive and negative edges at input I1 and between the negative edge and the next positive edge at input I1 must be greater than 2 ms.

#### Faulty measurement

The FM 352 returns the length value -1 in the event of errors in length measurement / edge detection.

A "length measurement" or "edge detection" may only perform up to 126 zero passes in one direction. The zero pass represents the rotary axis transition from the end of rotary axis value to 0 and vice versa. The FM 352 reports incorrect "length measurement" or "edge detection" if it detects more than 126 zero passes in one direction, regardless of whether or not it then detects any zero passes in the opposite direction.

A length measurement is also considered faulty if:

- the length measured at a rotary axis is greater than 2<sup>31</sup>.
- the on and off edges are detected simultaneously by the FM 352 (for example due to switch bounce).

This event can be signaled as a hardware interrupt.

#### Shift of the coordinate system during a length measurement

Conditions under which a shift of the coordinate system will influence the measured length:

- You are using an incremental encoder or an initiator or are operating the FM 352 in simulation mode.
- You are executing a "set reference point" or "retrigger reference point" function while a length measurement is active.

#### Example

To utilize the above mentioned influences on the measured length:

Your system always develops slip when you perform a length measurement.

The "Retrigger reference point" function can be used to correct this slip to allow correct length measurement values to be output.

4. Set the function switch in the channel DB.

##### Specifying software limit switches (S7-300, S7-400)

###### Software limit switch

Software limit switches only exist for the linear axis.

The operating range is the range limited by the software limit switches. The FM x52 can monitor the operating range limits.

The software limit switch start (SLS) must always be smaller than the software limit switch end (SLE).

The software limit switches are active when FM x52 is synchronized.

- When using an incremental encoder or initiator:

  The axis is initially not synchronized after each FM x52 startup. The parameterized software limit switches are not monitored until the axis is synchronized.
- When using an absolute encoder (SSI):

  The FM x52 is synchronized once it has received a complete frame without errors. The software limit switches are monitored from this time on. The absolute encoder you use must cover at least the operating range including the software limit switches.

###### Relationship between the operating range, encoder range and traversing range

- The operating range is the range which you specify for your job using the software limit switches.
- The encoder range represents the range explicitly covered by the encoder. For a linear axis, the module assigns the encoder range symmetrically across across the operating range. In other words, the module shifts the encoder range in order to equalize the distances between the software limit switches and the ends of the encoder range (see figure below).
- This traversing range is the value range the FM x52 can process. The traversing range is dependent on the resolution.

The following applies: Traversing range ≥ encoder range ≥ operating range

The following figure shows the relationship between software limit switches, the operating range, the encoder range and the traversing range.

![Relationship between the operating range, encoder range and traversing range](images/20383357323_DV_resource.Stream@PNG-en-US.png)

###### Absolute encoder adjustment procedure  (S7-300, S7-400)

###### Determining the correct absolute encoder adjustment

After initial parameter assignment, further steps must be taken to ensure that a correct relationship is created between the encoder and coordinate system. The procedure shown in the next section is based on the use of the parameter assignment dialogs.

1. Move the axis to a defined, reproducible point to which a unique coordinate is assigned.

   This can, for example, be the "software limit switch end".
2. Call the "[Set reference point](#set-reference-point-s7-300-s7-400)" setting using the coordinate defined in point 1.

   FM x52 now determines an encoder value for the reference point coordinate entered in the channel DB (REFPT in the channel DB), namely the absolute encoder adjustment. You can read this value with the "[Encoder data](#read-out-encoder-data-s7-300-s7-400)" setting.
3. Enter the value read in the "absolute encoder adjustment" field.
4. Save your parameter assignment to the corresponding parameter DB using the export function.
5. Download the data in the HWCN to the CPU.
6. Restart the CPU to activate the data.

> **Note**
>
> You perform this adjustment once during commissioning. Following parameter assignment, the FM x52 is synchronized during start-up as soon as a complete, faultless encoder frame is received after start-up.

###### Otherwise: Mechanical adjustment of the encoder

A correct relationship between the coordinate system and the encoder can be achieved as follows:

1. Traverse the axis to a defined, reproducible point to which a unique coordinate is assigned.

   This can, for example, be the "software limit switch start".
2. Enter this coordinate value in the machine data as "reference point coordinate".
3. Read the encoder value displayed at this position with the "[Encoder data](#read-out-encoder-data-s7-300-s7-400)" setting.
4. Enter this value in the machine data as the "absolute encoder adjustment".

After the parameter assignment a correct actual value will always be shown.

Instead of steps 3 and 4, you can also set the encoder to zero using "Reset", if available. Then enter the value "0" in the machine data as the absolute encoder adjustment.

###### Traversing range dependency on resolution

The traversing range is restricted by the numerical representation in FM x52. This numerical representation varies depending on the resolution. You should thus make sure that your specifications are always within the permitted limits.

The table below sets out the maximum traversing range:

| Resolution (RES) is in the range | Maximum traversing range |
| --- | --- |
| 0.1 µm/pulse ≦ RES < 1 µm/pulse | -10<sup>8 </sup>µm to 10<sup>8 </sup>µm (-100 m to +100 m) |
| 1 µm/pulse ≦ RES ≦ 1000 µm/pulse | -10<sup>9 </sup>µm to 10<sup>9 </sup>µm (-1000 m to +1000 m) |

###### Dependency between the velocity and resolution

The velocity indicated may vary within the following limits, depending on the resolution (data relate to mm units):

- from 1 µm/min to 90 m/min, at a resolution < 1 µm/pulse
- from 1 µm/min to 900 m/min, at a resolution ≥ 1 µm/pulse

The velocity value is calculated and filtered by the module at intervals of 4 ms.

Its minimum inaccuracy of one pulse/4 ms rules it out for closed-loop control.

#### Selecting monitoring (S7-300, S7-400)

##### Monitoring

**Wire break**

When its monitoring function is enabled, the FM x52 monitors the A, /A, B, /B, N, and /N signals of an incremental encoder. The monitoring function detects:

- Wire break
- Short circuits in individual cables

  For incremental encoders without a zero mark, you must either:

  - Disable the wire-break monitoring function
  - [Interconnect the N and /N signals externally](Cam%20control%20%28S7-300%2C%20S7-400%29.md#supported-encoders-s7-300-s7-400)
- Edge-to-edge distance of the count pulses
- Encoder supply failure

**Frame error**

You cannot switch off the monitoring of frame errors with absolute encoders (SSI). It monitors:

- the frame for start and stop bit errors
- the monoflop time of the connected encoder

**Missing pulses (incremental encoder)**

An incremental encoder must always return the same number of increments between two successive zero marks.

FM x52 checks whether the zero mark of an incremental encoder coincides with the correct encoder value.

Disable missing pulse monitoring for encoders without zero marks.

Also disable wire-break monitoring, or interconnect the zero mark inputs N and /N externally.

##### Specifying lead time (S7-300, S7-400)

###### Lead time

You can compensate any delays caused by the connected switchgear by setting a lead time. Define a lead time with a resolution of 100 µs. You can assign one lead time to each cam. The lead time applies to the cam start and cam end.

**Actuation distance**

The actuation distance of a cam is calculated continuously based on the current velocity and lead time. The entire cam is shifted in direction of the actual value by this value. The assigned range is the "static range", and the range calculated based on the lead time represents the "dynamic range".

Actuation distance = lead time x current velocity

FM x52 calculates the actuation distance of all cams within ¼ of the longest parameterized lead time. A very high lead time setting for a cam reduces the frequency of the dynamic adjustment calculation.

**Actual lead time**

To calculate the actual lead time:

1. Determine the cam cycle time: This is the time in which the FM x52 fully processes all cams. It depends on the number of cams parameterized.
2. Calculate the actual lead time based on the following formula:

   ![Lead time](images/26402722827_DV_resource.Stream@PNG-en-US.png)

   The identifiers have the following meanings:

   - Lead time <sub>act</sub> is the lead time set by the FM x52
   - Lead time <sub>d</sub> is your default setting.
   - Integer ( ) means that the values after the decimal point are ignored in the calculation of the parenthesis.

The maximum error of the lead time<sub>act</sub> is always less than [cam cycle time] x 4.

Example:

The following values are defined:

- Number of cams: Maximum 32 cams
- Cam cycle time: 40.96 µs
- Lead time<sub>d</sub> = 1000 µs

Result: An actual lead time of 983 µs.

> **Note**
>
> The actual is always less than the assigned lead time. It may be 0 even though the parameterized lead time ≥100 µs.
>
> The actuation distance of a rotary axis must be less than the rotary axis range and the inactive part of the cam. This must be guaranteed for all velocities.

> **Note**
>
> You may not implement a lead time in combination with an absolute encoder (SSI) and inverted parameterized count direction.

#### Triggering diagnostic interrupts (S7-300, S7-400)

##### Interrupt processing

The FM 352 can trigger hardware and diagnostic interrupts. You process these interrupts in an interrupt OB. If an interrupt is triggered without the corresponding OB being loaded, the CPU switches to STOP.

You enable the processing of diagnostic interrupts as follows:

1. Select the module in HWCN
2. Select "Edit" > Object Properties > Basic Parameters" to enable the diagnostic interrupt.
3. Save and compile the configuration.
4. Download the configuration to the CPU.

##### Overview of diagnostic interrupts

Events and errors which trigger a diagnostic interrupt:

- Operating error
- Machine data error
- Incorrect cam data
- Diagnostic errors

These errors are explained in detail in "[Error classes](#error-classes-s7-300-s7-400)".

##### Module response to an error with diagnostic interrupt

- Cam processing will be disabled.
- The synchronization will be cleared by the following diagnostic interrupts:

  - Front connector missing, no external auxiliary voltage
  - Zero mark error detected, cable fault (5 V encoder signals)
  - Traversing range exceeded (indicated by an operating error)
  - Set process value cannot be executed (indicated by an operating error).
- With one exception, control signals are no longer processed.

  Exception:

  When the software limit switches are overrun, a change of direction is still possible in simulation mode.
- Function switch and job processing continues.

##### Module detects an error ("incoming")

A diagnostic interrupt is an "incoming" event if at least one error is pending. If any errors are pending clearance, the queued errors are reported again as "incoming" events.

Sequence:

1. The FM 352 detects one or more errors and triggers a diagnostic interrupt. The error LED lights up. The error event is logged to the diagnostic buffer.
2. The CPU operating system calls the diagnostic interrupt OB, e.g. OB 82.
3. You can now evaluate the start information of OB82.
4. The OB82_MOD_ADDR parameter indicates which module triggered the interrupt.
5. For additional information, call the [CAM_DIAG](FM%20x52%20Cam%20Controlling%20%28S7-300%2C%20S7-400%29.md#cam_diag-s7-300-s7-400) instruction.

##### The module detects the transition to an error-free state ("outgoing")

A diagnostic interrupt is only registered "outgoing" if all errors on the module are cleared.

Sequence:

1. The FM 352 detects that all errors have been eliminated and triggers a diagnostic interrupt. The error LED goes out. The diagnostic buffer remains unchanged.
2. The CPU operating system calls the diagnostic interrupt OB, e.g. OB 82.
3. The OB82_MOD_ADDR parameter indicates which module triggered the interrupt.
4. Evaluate the OB82_MDL_DEFECT bit.

   When this bit is "0", no errors are present on the module. You can close the evaluation session at this point.

##### Diagnostic interrupt control by CPU states

- Diagnostic interrupts are blocked by FM 352 when the CPU is in STOP mode.
- If not all of the queued errors are eliminated while the CPU is in STOP mode, the FM 352 will report the remaining errors as "incoming" events again at the next CPU transition to RUN.
- If all queued errors are eliminated when the CPU is in STOP mode, the error-free state of the FM 352 will **not** be reported by a diagnostic interrupt after the transition to RUN mode.

#### Error evaluation in the user program (S7-300, S7-400)

##### Fault reaction in the user program

The user program allows a targeted response to errors. The following options are available:

- Return values (RETVAL) of the integrated instructions as a group display for errors that occurred during execution of the instructions.
- Error bits _ERR of the jobs as a group display for errors that occurred while a job was being processed.
- The DATA_ERR error bit as a group display for an error that the FM 352 has detected during a write job.
- The error identifier in JOB_ERR for the cause of an error in communication between instruction and FM 352.
- The [CAM_DIAG](FM%20x52%20Cam%20Controlling%20%28S7-300%2C%20S7-400%29.md#cam_diag-s7-300-s7-400) instruction for reading the diagnostic buffer of the FM 352. This is where you find out the causes of errors in jobs and asynchronous events (operator errors, diagnostic errors).
- Diagnostic interrupts for fast reaction to events.

#### Diagnostics buffer of the module (S7-300, S7-400)

##### Diagnostic events

The module´s diagnostic buffer contains a maximum of 4 diagnostic events and is organized as a ring buffer.

A diagnostic event is written in the buffer if an "incoming" (error) message is detected. This can be an alarm, a synchronous error (data error) or even an asynchronous error (operational errors and diagnostics errors). From a cause of error several entries can occur as follow-up errors. Outgoing messages do not create any entries in the diagnostic buffer.

For each diagnostic event the following is given:

- Status (always incoming)
- Internal error
- External error
- Error class
- Error number
- Channel number
- Cam number (for cam data errors)

When a diagnostic event is written in the diagnostic buffer, the checkback signal DIAG = 1 is set.

Using the [CAM_DIAG](FM%20x52%20Cam%20Controlling%20%28S7-300%2C%20S7-400%29.md#cam_diag-s7-300-s7-400) instruction, the diagnostic buffer can be transferred as a whole to a data block (diagnostic DB) or displayed by the error evaluation dialog of the configuration software. If the diagnostic buffer is read, the module sets the checkback signal DIAG = 0.

> **Note**
>
> If the diagnostic buffer is read simultaneously from the CAM_DIAG instruction and the error evaluation dialog, it is possible that a new diagnostic event not be detected by the program.
