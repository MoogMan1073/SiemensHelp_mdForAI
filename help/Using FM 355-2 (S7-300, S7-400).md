### Configuring the hardware (S7-300, S7-400)

#### Procedure

Configuration assumes that you have set up a project in which you can save the parameter assignment. The following briefly explains only the most important steps.

1. Start the TIA portal and open the device configuration.
2. Select one module rack.
3. Select one controller module from your module catalog.
4. Move the controller module to the appropriate place on the module rack.
5. From the configuration table, note the input address of the module, for example 272.

   The value that you read is displayed in decimal format.

### Module parameters (S7-300, S7-400)

#### Procedure

When assigning parameters you set the module parameters.

1. Double-click on the "Parameter setting" in the Device view of the module. In the Inspector window, select the "Properties" tab.

   **Result:** The properties are displayed in the inspector window.
2. Configure the basic parameters and the remaining parameters of the module.
3. Configure the remaining device parameters in the Design view.
4. Save the parameters entered with "Project&gt; Save".
5. Translate the project with "Edit &gt; Translate".
6. Load the configuration data with the CPU in STOP mode by selecting "Online &gt; Device load..."

   **Result:** The data stored in the memory of the CPU are then transferred directly to the module.
7. Carry out a CPU start-up.

3. Enter the following parameters:

4. Save the parameters entered with "Project&gt; Save".

#### Filter functions (S7-300, S7-400)

##### Procedure

To set the filter functions, proceed as follows:

1. Double-click on the "Parameter setting" in the device view of the module.
2. Navigate in the design view to "Analog input &gt; input [No]&gt; Filter functions".
3. Enter the following parameters:

   - To filter on a time constant you should enable the "signal filter" option.

     If the "signal filter" option is selected, the time constant filter can be set.
   - Filter time constant

     Can only be set if the "signal filter" option is enabled.
   - To linearize the encoder signal you should activate the "square root filter" option.
   - To linearize the encoder signal characteristics you should activate the "polygon filter" option.
4. Save the parameters entered with "Project&gt; Save".

##### Signal filter

Filter 1. order, whose time behavior is set through the time constants.

##### Square root filter

Used to linearize encoder signals in which the actual value is present as a physical quantity, which is square in relation to the measured process variable.

##### Polygon filter

Used to linearize encoder characteristics.

The 13 selectable points are selectable in:

- mA at current input
- mV with voltage input

##### Scaling

To convert the input signal into another physical unit through linear interpolation between the initial value (low value) and the ending value (high value).

#### Implementing a constant or a cascade controller (S7-300, S7-400)

##### Constant or cascade controller

In the controller structure of constant or cascade controllers the effective current value is equal to the actual value A.

##### Selecting the controller type

To implement a constant or cascade controller, proceed as follows:

1. Double-click on the "Parameter setting" in the device view of the module.
2. Navigate in the design view to "Controller&gt; channel [No]".
3. In the "Encoder type" section select the "Constant or Cascade controller".
4. Save the parameters entered with "Project&gt; Save".

##### Processing setpoints

To edit setpoints, proceed as follows:

1. Double-click on the "Parameter setting" in the device view of the module.
2. Navigate in the design view to "Controller&gt; Channel [No]&gt; Deviation".
3. Set in the different areas the corresponding parameters:

   - Setpoint source

     Select the source of the setpoint.
   - Safety setpoint

     Set your safety setpoints here. Set also the response to a CPU fault and the startup behavior.
   - Setpoint ramp

     Set the start-up time.
   - Setpoint limit

     Select the high and low setpoint limits.
4. Save the parameters entered with "Project&gt; Save".

##### Selecting a signal source for actual value A

To select the signal source for the actual value A, proceed as follows:

1. Double-click on the "Parameter setting" in the device view of the module.
2. Navigate in the design view to "Controller&gt; Channel [No]&gt; Deviation".
3. In the "Control value" section select the corresponding source. The following choices are available:

   - Zero
   - Processed analog input
4. Save the parameters entered with "Project&gt; Save".

#### Implementing three component controllers (S7-300, S7-400)

##### Three component controller

In the controller structure of a three component controller the effective actual value is obtained by summing the three actual values of A, B and C and then adding an adjustable offset. The actual values of B and C can also be evaluated via factors.

##### Selecting the controller type

Proceed as follows to implement a three-term controller:

1. Double-click on the "Parameter setting" in the device view of the module.
2. Navigate in the design view to "Controller&gt; channel [No]".
3. In the "Controller type" section select the "Constant or Cascade controller" option.
4. Save the parameters entered with "Project&gt; Save".

##### Processing setpoints

To adjust the setpoint processing mode, proceed as follows:

1. Double-click on "Parameter setting" in the device view of the module.
2. Navigate in the design view to "Controller&gt; Channel [No]&gt; Deviation".
3. In the "Setpoint source" section select the source from which the setpoint shall be obtained.
4. In the "Safety setpoint" section enter a value to be utilized upon CPU fault and start-up as a safety value.
5. Select the behavior of the controller module considering the values of the "Reaction to CPU errors" and the "Start-up behavior" options. You may select amongst the following options:

   - Setpoint = last valid setpoint: the setpoint will be assigned the last valid setpoint upon occurrence of the event. The last valid setpoint will remain unchanged.
   - Setpoint = safety setpoint: the setpoint will be assigned the safety setpoint upon the occurrence of the event. Thus, a secured emergency operation can be performed.
6. Set in the "setpoint ramp" the rise time which shall be met upon a setpoint change to reduce overshoot.
7. In the "Setpoint limits" section set both the high and a low limits for the setpoint.
8. Save the parameters entered with "Project &gt; Save".

##### Selecting a signal source for actual values A, B. and C

To set the signal sources for the actual values of A, B and C, proceed as follows:

1. Double-click on "Parameter setting" in the device view of the module.
2. Navigate in the configuration view to "Controller &gt; Channel [No] &gt; System deviation".
3. In the "Control value" section select the signal source for each actual value. Here you can select whether you use no signal ("zero") as a source or one of the processed analog inputs.
4. Save the parameters entered with "Project &gt; Save".

##### Registering the factors for actual values B and C and the offset value

Proceed as follows to set the signal sources for the actual values A, B and C:

1. Double-click on "Parameter setting" in the device view of the module.
2. Navigate in the configuration view to "Controller &gt; Channel [No] &gt; System deviation".
3. In the"Sum" section set the necessary weight factors for the actual values B and C.
4. Set the required "offset" value that should be added to the actual values.
5. Save the parameters entered with "Project &gt; Save".

#### Implementing ratio or mixture controllers (S7-300, S7-400)

##### Selecting the controller type

Proceed as follows to implement a ratio or blending controller:

1. Double-click on "Parameter setting" in the device view of the module.
2. Navigate in the configuration view to "Controller&gt; channel [No]".
3. In the "Encoder type" section select the "Ratio or Mixture controller".
4. Save the parameters entered with "Project &gt; Save".

##### Processing setpoints

Proceed as follows to adjust the setpoint processing mode:

1. Double-click on "Parameter setting" in the device view of the module.
2. Navigate in the configuration view to "Controller &gt; Channel [No] &gt; System deviation".
3. In the "Setpoint source" section, select the source from which the setpoint is to be obtained.
4. In the "Safety setpoint" section, enter a value to be utilized upon CPU fault and start up as a safety value.
5. Select the behavior of the controller module for "Reaction to CPU errors" and "Start-up behavior". You may select from amongst the following options:

   - Setpoint = last valid setpoint: the setpoint will be assigned the last valid setpoint upon occurrence of the event. The last valid setpoint will remain unchanged.
   - Setpoint = safety setpoint: the setpoint will be assigned the safety setpoint upon the occurrence of the event. This enables secured emergency operation.
6. In the "setpoint ramp" section, set the rise time to be adhered to upon a setpoint change to reduce overshoot.
7. In the "Setpoint limits" section, set both an high and a low limit for the setpoint.
8. Save the parameters entered with "Project &gt; Save".

##### Selecting the signal source for actual values A and D

To set the signal sources for the actual values of A and D, proceed as follows:

1. Double-click on "Parameter setting" in the device view of the module.
2. Navigate in the configuration view to "Controller &gt; Channel [No] &gt; System deviation".
3. Select the signal source for each actual value in the "Manipulated value" section. You can choose whether using a no signal ("zero") as a source or one the processed analog inputs. For the actual value for D, you may also select the controller control value.
4. Save the parameters entered with "Project &gt; Save".

##### Registering Multipliers

Proceed as follows to set the signal sources for actual values A and D:

1. Double-click on "Parameter setting" in the device view of the module.
2. Navigate in the configuration view to "Controller &gt; Channel [No] &gt; System deviation".
3. In the "Multiply" section select the desired offset.
4. Save the parameters entered with "Project &gt; Save".

For ratio controllers the actual value A is used as control variable while the actual value D stores the ratio magnitude. The setpoint input value is used as a ratio factor. Multiplying it by the actual value of D and adding the adjustable offset will yield the effective setpoint value. If actual value D is off only the offset should be added to the setpoint.

#### Setting the alarm generation (S7-300, S7-400)

##### Procedure

Proceed as follows to set interrupt generation:

1. Double-click on "Parameter setting" in the device view of the module.
2. Navigate in the configuration view to "Controller &gt; Channel [No] &gt; System deviation".
3. In the "Alarm generation" section select the necessary options:

   - In the "Limit monitor for" section enter the signal for which the alarm should be set.
   - In "Alarm high limit", set the high limit beyond which the alarm should be triggered.
   - In "Warning high limit", set the value beyond which the "approaching high limit" warning should be issued.
   - In "Warning low limit", set the value below which the "approaching low limit" warning should be issued.
   - In "Alarm low limit", set the low limit below which the alarm should be triggered.
   - To avoid flickering of the monitor you may set the hysteresis parameter in "hysteresis".
4. Save the parameters entered with "Project &gt; Save".

#### Switching disturbances (S7-300, S7-400)

##### Procedure

To switch a disturbance, proceed as follows:

1. Double-click on the "Parameter setting" in the device view of the module.
2. Navigate in the design view to "Controller&gt; channel [No]&gt; Controller algorithm".

5. Save the parameters entered with "Project&gt; Save".

#### Reversing the controller effect (S7-300, S7-400)

##### Principle

Controller reversal may be achieved by setting a negative proportional gain thus moving from "increasing deviation value = increasing correction value" to "increasing deviation value = decreasing correction value".

##### Procedure

To reverse the controller effect, proceed as follows:

1. Double-click on the "Parameter setting" in the device view of the module.
2. Navigate in the design view to "Controller&gt; channel [No]&gt; Controller algorithm".
3. In the "General" section, set a proportional gain of opposite sign.

   As an example, if the current setting shows a positive proportional gain and you wish to reverse the controller effect, set a negative proportional gain.
4. Save the parameters entered with "Project&gt; Save".

### Options of controller tuning (S7-300, S7-400)

#### Pretuning

During this tuning, the working point is approached from the cold state through a setpoint jump.

With TUN_ON = TRUE, you can establish the tuning readiness. The controller switches from PHASE = 0 to PHASE = 1.

![Pretuning](images/31421618315_DV_resource.Stream@PNG-en-US.png)

The tuning manipulated variable (LMN0 + TUN_DLMN) is activated by a setpoint change (transition PHASE 1 -&gt; 2). The setpoint is not effective until the inflection point has been reached (automatic mode is not enabled until this point is reached).

The user is responsible for defining the output excitation delta (TUN_DLMN) according to the permitted process value change. The sign of TUN_DLMN must be set depending on the intended process value change (take into account the direction in which the control is operating).

The setpoint step change and TUN_DLMN must be suitably matched. If the value of TUN_DLMN is too high, there is a risk that the point of inflection will not be found before 75% of the setpoint step change is reached.

TUN_DLMN must still be high enough to ensure that the process value reaches at least 22% of the setpoint step change. Otherwise, the process will remain in tuning mode (phase 2).

Solution: Reduce the setpoint value during the inflection point search.

> **Note**
>
> If processes are extremely sluggish, it is advisable to specify a target setpoint that is somewhat lower than the desired operating point during tuning and to monitor the status bits and PV closely (risk of overshooting).
>
> **Tuning only in the linear range:**
>
> The signals of certain processes (e.g., zinc or magnesium smelters) will pass a non-linear area at the approach of the operating range (change in the state of aggregation).
>
> By selecting a suitable setpoint step change, tuning can be limited to the linear range. When the process value has passed 75% of the setpoint step change (SP_INT-PV0), tuning is ended.
>
> At the same time, TUN_DLMN should be reduced to the extent that the point of inflection is guaranteed to be found before 75% of the setpoint step change is reached.

#### Fine tuning heating

During this tuning, the process with a constant setpoint is activated through a output value jump.

When you modify the setpoint value, the new value will not take effect until the point of inflection has been reached (automatic mode will not be enabled until this point has been reached).

The user is responsible for defining the output excitation delta (TUN_DLMN) according to the permitted process value change. The sign of TUN_DLMN must be set depending on the intended process value change (take into account the direction in which the control is operating).

#### Fine tuning cooling

During this tuning, the process with a constant setpoint is activated through a output value jump.

![Fine tuning cooling](images/31417952267_DV_resource.Stream@PNG-en-US.png)

The user is responsible for defining the output excitation delta (TUN_CLMN) according to the permitted process value change. The sign of TUN_CLMN must be set depending on the intended process value change (take into account the direction in which the control is operating). Note that a negative TUN_CLMN causes an increase in cooling power. But you can also use TUN_CLMN &gt; 0.0 to activate a setpoint jump by reducing the cooling power.

#### Process requirements (S7-300, S7-400)

##### Transient response

The process must have a stable, asymptotic transient response with time lag.

The actual value must settle to steady state after a jump of the manipulated variable. This therefore excludes processes that already show an oscillating response without control, as well as processes with no compensation (integrator in the control system).

##### Guaranteeing a stationary initial state (phase 0)

With lower-frequency oscillations of the actual value, for example, due to incorrect controller parameters, the controller must be put in manual mode before the optimization is started and wait for the oscillation to stop. You can also switch to a PI controller with small proportional gain and high integration time.

Now you have to wait until the stationary state is reached, this means, until the process value and output value have a steady state. It is also permissible to have an asymptotic transient oscillation or slow drifting of the process value (stationary state, see the following image). The output value must be constant or fluctuate by a constant average.

> **Note**
>
> Avoid changing the manipulated variable shortly before starting the tuning. A change of the manipulated variable can occur in an unintended manner through the establishment of the test conditions (for example, closing an oven door)! If this does happen, you have to at least wait until the process value has an asymptotic transient oscillation in a stationary state again. Better controller parameters can be reached if you wait until the transient effect has completely subsided.

In the following image, the transient oscillation is illustrated in the stationary state:

![Guaranteeing a stationary initial state (phase 0)](images/31421923211_DV_resource.Stream@PNG-en-US.png)

##### Linearity and operating range

The process response must be linear across the operating range. Non-linear response occurs, for example, when a state of aggregation changes. Optimization must take place in a linear part of the operating range.

This means, during optimization and normal control operation non-linear effects within the operating range must be insignificant. It is, however possible to retune the process when the operating point changes, providing optimization is repeated in the close vicinity of the new working point and non-linearity does not occur during the optimization.

If a specific static non-linearity (e.g. valve characteristics) is known, it is always advisable to compensate this with a polyline to linearize the process response.

##### Disturbance in temperature processes

Disturbances such as the transfer of heat to neighboring zones must not affect the overall temperature process too much. For example, when optimizing the zones of an extruder, all zones must be heated simultaneously.

#### Process of controller tuning (S7-300, S7-400)

##### PI/PID controller parameters

The FM 355-2 auto-tuning feature automatically sets the PI/PID controller parameters. You can tune the heating and cooling processes as well as split-range processes with two counteracting final controlling elements (e.g. the final controlling elements for heating and cooling processes).

##### Phases of controller tuning

For the controller tuning, individual phases are run through, which you can read on the parameter PHASE .

##### PHASE = 0

Tuning is not running. The FM 355-2 works in automatic or manual mode.

During PHASE = 0, you can make sure that the controlled system fulfills the requirements for tuning.

At the end of tuning, FM 355-2 switches back into PHASE = 0.

##### PHASE = 1

FM 355-2 is ready for tuning. PHASE = 1 may only be started if the requirements for tuning are fulfilled.

During PHASE = 1, the following values are determined:

- Process value noise NOISE_PV
- Initial slope PVDT0
- Average of the manipulated variable

##### PHASE = 2

In phase 2, the process value attempts to detect the point of inflection with a constant manipulated variable. This method prevents the point of inflection from being found too early as a result of process variable noise.

Initially, this averaging is inactive; in other words, averaging always takes place over one cycle. As long as the noise exceeds a certain level, the number of cycles is doubled.

The period and amplitude of the noise are calculated. The search for the point of inflection is canceled and phase 2 is exited only when the gradient is always smaller than the maximum rise during the estimated period. TU and T_P_INF are, however, calculated at the actual point of inflection.

Tuning, however, is only ended when the following two conditions are met:

1. The process value is more than 2*NOISE_PV away from the point of inflection.
2. The process value has exceeded the point of inflection by 20%.

   > **Note**
   >
   > When exciting the process using a setpoint step change, tuning is ended at the latest when the process value exceeds 75% of the setpoint step change (SP_INT-PV0) (see below).

##### PHASE = 3, 4, 5, 6

The phases 3, 4, 5 and 6 last 1 cycle each.

In Phase 3, the valid PI/PID parameters are saved before the optimization and the process parameter is calculated.

In Phase 4, the new PI/PID parameters are calculated.

In phases 5 and 6, the new manipulated variable is calculated and the controlled system is given.

If a controlled system of type I was identified, tuning mode stops and tuning is once again in phase 0. The controller now starts with LMN = LMN0 + 0.75*TUN_DLMN always in automatic mode (even if you controlled in manual mode before you started tuning).

If a controlled system of type II or III was identified, tuning changes to phase 7.

##### PHASE = 7

There is a risk with typical temperature systems (process type I) that the point of inflection is detected too early as a result of noise. The lower point of inflection time T_P_INF may possibly result in the detection of a process type II or III.

There is a check in phase 7 to see if the process type is correct. This testing occurs in the automatic mode with the recently recalculated controller parameters and ends at the latest 6*TA (equilibrium time) after the point of inflection. If process type I is detected, the controller parameters are recalculated (STATUS_D = 122) otherwise they remain the same.

The checking of the process type is cancelled, if the process value during tuning at the operating point reaches the process value once again prior to tuning PV0.

If phase 7 is canceled with TUN_ON = FALSE, the already determined controller parameters are maintained!

> **Note**
>
> It is generally possible to start the heat tuning during cooling. The order check in phase 7 will not take place in this case.
>
> The reverse case (starting cooler tuning during heating) is not critical, because a cooler tuning never runs through phase 7!

##### Premature cancellation of the optimization

In Phase 1, 2 or 3, you can cancel the optimization by resetting TUN_ON = FALSE without calculating new parameters. The controller starts in automatic mode with LMN = LMN0. If the controller was in manual mode before the tuning, the old manual manipulated variable is output.

If tuning is canceled in Phase 4, 5, 6 or 7 with TUN_ON = FALSE, the controller parameters determined until then are maintained.

##### Optimization result

The left number of STATUS_H and STATUS_C displays the tuning state.

| STATUS_H STATUS_C | Result |
| --- | --- |
| 0 | Default or new controller parameters have not yet been found. |

- Factor for the attenuation of the proportional component PFAC_SP = 0.8
- Controller GAIN

##### Point of inflection not reached (only if excited by setpoint step change)

At the latest, tuning is ended when the process value has passed 75% of the setpoint step change (SP_INT-PV0). This is signaled by "inflection point not reached" in STATUS_H/STATUS_C (2xx2x).

The currently valid setpoint always applies. By reducing the setpoint, it is possible to achieve an earlier end of the tuning function.

In typical temperature processes, cancelation of tuning at 75% of the setpoint step change is normally adequate to prevent overshoot. However, **caution** is advised, particularly in processes with a greater delay (TU/TA &gt; 0.1, process type III). If manipulated variable excitation is too strong compared to the setpoint step change, the process value can overshoot heavily (up to a factor of 3).

In higher-order processes, if the point of inflection is still a long way off after reaching 75% of the setpoint step change, there will be significant overshoot. In addition, the controller parameters are too stringent. In this case, you should reduce the controller parameters or repeat the attempt.

The following schematic illustrates the overshoot of the process variable when the excitation is too strong (process type III):

![Point of inflection not reached (only if excited by setpoint step change)](images/31434709771_DV_resource.Stream@PNG-en-US.png)

In typical temperature processes, cancelation shortly before reaching the point of inflection is not critical in terms of the controller parameters.

If you repeat the attempt, reduce TUN_DLMN or TUN_CLMN or increase the setpoint step change.

Principle: The value of the manipulated variable used for tuning must be suitable for the setpoint step change.

##### Error estimating the delay time or order

The delay time (STATUS_H/STATUS_C = 2x1xx, 2x2xx or 2x3xx) or order (STATUS_H/STATUS_C = 21xxx or 22xxx) were not acquired correctly. Operation continues with an estimate that can lead to non-optimum controller parameters.

Repeat the tuning procedure and ensure that disturbances do not occur at the process value.

> **Note**
>
> The special case of a PT1-only process is also indicated by STATUS_H/STATUS_C = 2x2xx (TU &lt; CYCLE). In this case, it is not necessary to repeat the attempt. Reduce the controller parameters if the control oscillates.

##### Quality of measuring signals (measurement noise, low-frequency interference)

The results of tuning can be distorted by measurement noise or by low-frequency interference. Note the following:

- During one noise period, the process value should be sampled at least twice. The degree of noise should not exceed 5% of the useful signal change.
- High-frequency interference cannot be filtered out by a software block. This should be filtered earlier in the measuring sensor to prevent the aliasing effect.

  The following schematic illustrates the aliasing effect when the sampling time is too long:

  ![Quality of measuring signals (measurement noise, low-frequency interference)](images/31434798475_DV_resource.Stream@PNG-de-DE.png)
- In case of low-frequency interferences, we assume that the sampling time of the FM 355-2 is small enough. However, the FM 355-2 must then generate a uniform measuring signal by having a large interval in the mean value filtering. Mean value filtering must extend over at least two noise periods. Internally in the block, this soon results in higher sampling times such that the accuracy of the tuning is adversely affected. Adequate accuracy is guaranteed with at least 40 noise periods up to the point of inflection. Possible remedy when repeating the attempt: Increase TUN_DLMN/TUN_CLMN.

##### Overshoot

Overshoot can occur in the following situations:

| Situation | Cause | Remedy |
| --- | --- | --- |
| End of tuning | - Excitation by a too high manipulated value change compared with the setpoint step change (see above). - PI controller activated by PID_ON = FALSE. | - Increase the setpoint step change or reduce the manipulated value step change. - If the process permits a PID controller, start tuning with PID_ON = TRUE. |

#### Tuning with step controller (S7-300, S7-400)

##### Introduction

The general information on controller optimization applies.

##### Peculiarities of step controllers

- Step controllers on the FM 355-2 operate without control zone.
- Phase 7 is not executed.
- No cooling optimization.

  > **Note**
  >
  > The motor transition time is not determined by tuning and must be measured or set before you start tuning.

##### Controller design

The motor actuating time MTR_TM should be as small as possible compared to the inflection point time T_P_INF and the equivalent dead time TU.

Controllers with a softer action are automatically generated for longer motor actuating times.

The higher the process excitation TUN_DLMN, the higher is the influence of the motor actuating time on the controller design.

##### PI or PID parameters

PI parameters are attenuated by 25% compared to a controller designed for closed-loop controllers and pulse controllers. PID parameters are also determined (but without the 25% safety margin). The PID parameters should only be used if the motor actuating time is not too high compared to the process parameters and when the load on the final controlling element stays within the hysteresis due to the derivative component.

##### Step controller without position feedback

The output of Open/Close instructions is stopped right at the start of phase 1 (quasi manual mode).

The mean value of the manipulated variable in phase 1 (LMN0) will not be calculated.

At the start of phase 2 and during the time MTR_TM * TUN_DLMN / 100, an Open instruction (or with negative TUN_DLMN, a close instruction) is output.

When an end stop signal is triggered during the pulse action in phase 2, the effective TUN_DLMN is calculated as: 100 * time / MTR_TM.

When the value of the effective TUN_DLMN &lt; 5%, the error message STATUS_H=30002 is output and optimization is canceled. This error can only occur if the final controlling element is unexpectedly close to a limit.

##### End or cancellation of optimization

The controller starts with LMN = LMN0 + TUN_DLMN.

#### Parallel tuning of controller channels (S7-300, S7-400)

##### Neighboring zones with strong temperature coupling effect

When two or multiple controllers of a module are used to control the temperature, for example, on a plate (e.g. two heaters and two measured actual values with strong temperature coupling), you have the option of parallel controller tuning.

##### Procedure

Follow these steps to tune several channels at once:

1. Configure the channel group in Zone A or Zone B.
2. Start tuning in one of these channels. Tuning of the other channels will then be started automatically.

**Note**

**Tuning is aborted**

Each participating controller will output its tuning manipulated variable until all controllers have exited phase 2. This avoids that the controller that completes tuning first corrupts the tuning result of the other controller due to the change in its manipulated variable.

### Manual fine tuning in control mode (S7-300, S7-400)

#### Introduction

The following measures can be employed in order to achieve a non-overshooting response to setpoint changes:

#### Adapt control zone

During tuning, the FM 355-2 determines the control zone width CON_ZONE and, with an appropriate process type (process type I and II), a PID controller is activated: CONZ_ON = TRUE. During control mode, you can modify the control zone or switch it off completely (with CONZ_ON = FALSE).

**No control zone for process type III, PI controller, step controller**

Activating the control zone with higher order processes (process type III) does not normally bring any benefit since the control zone is then larger than the control range that can be achieved with a 100% manipulated variable. There is also no advantage in activating the control zone for PI controllers.

> **Note**
>
> Before you switch on the control zone manually, make sure that the control zone width is not too small. This means, the manipulated variable and actual value will oscillate if the control zone width is too small.

#### Weakening control response continuously with PFAC_SP

The control zone offers the best dynamic option of achieving a control response that is free of overshoot. Use it wherever possible, so for PID controllers for process types I and II.

For PI controllers, process type III or setpoint jumps within the control zone you can weaken the control response with parameter PFAC_SP. This parameter specifies the amount of P action that is affective for setpoint jumps.

Regardless of the process type, PFAC_SP is set to a default value of 0.8 by the tuning function; you can later modify this value if required. To limit overshoot during setpoint jumps (with otherwise correct controller parameters) to approximately 2%, the following values are adequate for PFAC_SP:

|  | Process type I | Process type II | Process type III |
| --- | --- | --- | --- |
|  | Typical temperature process | Intermediate range | Higher order temperature process |
| PI | 0.8 | 0.82 | 0.8 |
| PID | 0.6 | 0.75 | 0.96 |

Adapt the default factor (0.8) especially under the following situations:

- Process type I with PID (0.8 →0.6): With PFAC_SP = 0.8, setpoint jumps will still lead to approx. 18% overshoot.
- Process type III with PID (0.8 →0.96): With PFAC_SP = 0.8, the setpoint jumps is attenuated too heavily. This leads to a significant loss of tuning time.

#### Attenuation of control parameters

When a closed-loop control circuit oscillates or if overshoot occurs after setpoint jumps, you can reduce the controller’s GAIN (e.g. to 80% of the original value) and increase the integration time TI (e.g. to 150% of the original value). If the analog manipulated variable of the closed-loop controller is converted to binary actuating signals by a pulse shaper, quantization noise may cause minor permanent oscillation. You can eliminate this effect by increasing the dead band width DEADB_W.

### Saving and retrieving controller parameters (S7-300, S7-400)

#### Back-up file records

The FM 355-2 features an effective parameter record system as well as a backed up parameter record system. Save and retrieval operations are performed by means of SAVE_PAR or UNDO_PAR in OP structure of the FMT_PID instruction.

For example, prior to a manual change you can save the current parameters via SAVE_PAR = TRUE. You can retrieve the last saved controller parameters via UNDO_PAR = TRUE and enable them for the controller. SAVE_PAR or UNDO_PAR is reset by FMT_PID at the end of this action.

> **Note**
>
> At the end of controller tuning, the saved parameters are overwritten with parameter data that were valid prior to tuning (Exception: this intermediate saving operation is discarded in a cooling tuning process).

#### Change between PI and PID parameters

After tuning, the PI and PID parameters are saved on the FM355-2. You load these parameter records by setting FMT_PID.OP.LOAD_PID. If PID_ON in the structure PAR = TRUE, the PID parameter record is copied to the effective controller parameters; otherwise, the PI parameter record is copied. LOAD_PID is reset by FMT_PID at the end of this action. Of course, the value of PID_ON is retained, as it is not an operator control parameter.

|  | PI parameter record | PID parameter record |
| --- | --- | --- |
| Controller GAIN | PI_GAIN | PID_GAIN |
| Integration time | PI_TI | PID_TI |
| Derivative time | 0.0 | PID_TD |

> **Note**
>
> Please note the following:
>
> - Controller parameters are only written back with UNDO_PAR or LOAD_PID if the controller gain (SAV_GAIN, PID_GAIN or PI_GAIN) is not equal to zero:
> - This strategy takes into account the situation that no tuning has yet been made or that PID parameters are missing. For example, if PID_ON was TRUE and PID_GAIN = 0.0, PID_ON is set to FALSE and the PI parameters will be copied.
> - D_F, PFAC_SP are set to default values by the tuning. The user can later modify them. LOAD_PID does not modify these parameters.
> - With LOAD_PID, the control zone is always recalculated (CON_ZONE = 250/GAIN or 375/GAIN if RATIOFAC &lt;&gt; 0.0), irrespective of the status CONZ_ON = FALSE.

#### Pulse-free changeover with LOAD_PID and UNDO_PAR

The changeover is performed so that the sum of the proportional action and integral action stays equal. A changeover from PID to PI triggers a jump of the manipulated variable, as the derivative action is disabled. Initially, the changeover from PI to PID is pulse-free. After a few cycles, however, the derivative action will lead to a fast change.

### Measuring motor actuating time (S7-300, S7-400)

#### Introduction

The motor transition time has to be as accurate as possible for good controller results. The data in the actuator documentation contains average values for this type of actuator. The value for the specific actuator used may differ.

You can measure the motor transition time during commissioning if you are using actuators with position feedback or endstop signals. The motor transition time cannot be measured if neither position feedback nor endstop signals are available.

#### Requirement

- The configured motor transition time must be half the size of the real transition time. The actuator is moved for a maximum of twice the transition time in one direction.
- The position feedback must be interconnected for actuators with analog position feedback.
- The endstop signals must be interconnected for actuators with endstop signals.

#### Actuators with analog position feedback

Proceed as follows to measure motor transition time with position feedback:

1. Select the instance DB for which you want to measure the motor transition time.

   The current position feedback (starting position) will be displayed. Click the ![Actuators with analog position feedback](images/23498796811_DV_resource.Stream@PNG-de-DE.png) "Start transition time measurement" icon.

#### Result

The actuator is moved from the starting position to the target position. Time measurement starts immediately and ends when the actuator reaches the target position. The motor transition time is calculated according to the following equation:

Motor transition time = (output value high limit – output value low limit) × Measuring time / AMOUNT (target position – starting position).

The progress and status of transition time measurement are displayed. The measured transition time is displayed in the field "Measured transition time". Once transition time measurement is complete, the manual mode of the transition signals is switched on (OP.LMNS_ON = TRUE). The actuator will stop being moved.

The actuator is moved first to the low endstop, then to the high endstop and then back to the low endstop. Time measurement will start once the actuator has reached the lower endstop and will end when the actuator reaches the lower endstop for the second time. The motor transition time is equal to the time measured divided by two.

### Triggering diagnostic interrupts (S7-300, S7-400)

#### What is a diagnostic interrupt?

If the user program is to react to an internal fault or external error, you can configure a diagnostic interrupt. The cyclical CPU program is interrupted and the diagnostic interrupt OB, OB 82 is called.

#### Which events may trigger a diagnostic interrupt?

The list shows you which events can trigger a diagnostic interrupt:

- Module configuration missing or faulty
- Module defective
- Wire break when analog inputs are made (4 up to 20 mA only)
- High limit violated and low limit violated during analog inputs
- Load break or short-circuit during analog outputs

#### Enabling the diagnostic interrupt

You can enable or disable the diagnostic interrupt for the module in the basic parameter tab.

The default setting for the diagnostic interrupt is disabled.

#### Reactions to an event triggering an interrupt

If an event occurs that can trigger a diagnostic interrupt, the following occurs:

- The diagnostics information is written to the diagnostics data records DS0 and DS1 on the module.
- The group error LED lights up.
- Diagnostic interrupt OB is called (OB 82).
- Diagnostic data record DS0 is written to the start information of diagnostic interrupt OB.
- If there is no hardware defect, the module resumes control operation.

If no OB 82 is programmed, the CPU goes into STOP.

#### Diagnostics data record DS0 and DS1

The information about which event triggered a diagnostic interrupt is written to the diagnostic data records DS0 and DS1. The length of the diagnostic data record DS0 is four bytes; the record length of DS1 is 16 bytes of which the first 4 bytes are identical with DS0.

### Diagnostic data records DS0 and DS1 (S7-300, S7-400)

#### Read data record from the module

The diagnostic data record DS0 is automatically transferred to the start information when the diagnostic OB is called. There these four bytes are stored in the local data (byte 8 - 11) of the OB 82.

The diagnostic data record DS1 (and hence also the content of the DS0) can be read from the module by means of FB 55 "FMT_DS1". This only makes sense if an error is reported in a channel in the DS0.

The FB 55 "FMT_DS1" must be called in the same OB as an FB FMT_PID that might be present (e.g. OB 35). You can do this as follows: While OB 82 is being processed, set the READ_DS1 bit. The FMT_DS1 in OB 35 then reads the diagnostic data record DS1.

#### How does the diagnosis text appear in the diagnostics buffer?

If you want to enter the diagnostics message in the diagnostics buffer, call the "WR_REC" instruction "Enter user-specific message in the diagnostics buffer" in the user program. The event number of the respective diagnosis message is specified on the input parameter EVENTN. The interrupt is entered in the diagnostics buffer defined as x = 1 for incoming and x = 0 for outgoing. In addition to the time of the entry, the appropriate diagnosis text is displayed in the "Meaning" column.

#### The assignment of the diagnostic data records DS0 and the Start information

The following table shows the assignment of the diagnostic data record DS0 in the Start information. Any bits not listed have no meaning and are zero.

Assignment of diagnostic data record DS0

| Byte | Bit | Meaning | Remark | Event no. |
| --- | --- | --- | --- | --- |

### Measuring transducer fault (S7-300, S7-400)

#### Measuring transducer faults

The FM 355-2 can identify the following measuring transducer faults:

- Measuring range violation (underrange)
- Measuring range violation (overrange)
- Wire break (not for all measuring ranges)

If one of these faults occur, the group error bit "external error" is set in diagnostic data record DS0 and channel-specific error bits are set in diagnostic data record DS1 (see table above). When these faults disappear, the respective bits are reset.
