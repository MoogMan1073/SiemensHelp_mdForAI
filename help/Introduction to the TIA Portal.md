## Starting and exiting the TIA Portal

### Starting the TIA Portal

To start the TIA Portal, follow these steps:

1. In the Windows start menu, select the command "Siemens Automation &gt; TIA Portal V18".

   The TIA Portal opens with the last settings used.

### Exiting the TIA Portal

To exit the TIA Portal, follow these steps:

1. In the "Project" menu, select the "Exit" command.

   If the project contains any changes that have not been saved, you will be asked if you wish to save them.

   - Select "Yes" to save the changes in the current project and close the TIA Portal.
   - Select "No" to close the TIA Portal without saving the most recent changes in the project.
   - Select "Cancel" to cancel the closing procedure. The TIA Portal will remain open if you select this option.

### Overview of the settings for diagnostics and feedback

#### Overview

The following table shows the settings that you can make for diagnostics and feedback:

| Group | Setting | Description |
| --- | --- | --- |
| Diagnostics and feedback | Activate feedback on diagnostic data | When the setting is enabled, diagnostic data is generated and saved on the computer.  Diagnostic data is used to keep the TIA Portal secure and current, to detect problems, and for product improvements.  You can only transfer this data as a crash report by email if the TIA Portal detects problems.  The diagnostic data is saved in the following directory:   ...\ProgramData\Siemens\ETWEventCollector |
| Activate collection of advanced diagnostic data | When the setting is enabled, advanced diagnostic data is generated and saved on the computer.   Extended diagnostic data supports precise troubleshooting in the TIA Portal.  You can transmit this data as a crash report by email.  The advanced diagnostic data is saved in the following directory:  ...\ProgramData\Siemens\Automation\Diagnostics\Dumps |  |
| Activate feedback on statistical usage data | When the setting is enabled, statistical usage and diagnostics data is generated and saved on the computer. The statistical usage and diagnostic data is saved in the following directory:  ...\ProgramData\Siemens\Automation\TelemetryConnector\EventPersistence  In a second step, the statistical usage and diagnostic data is transferred to Siemens via secure communication for the purposes listed above.   When the setting is disabled, no data is collected on the computer and transferred to Siemens (except for the message to disable the setting).  When the setting is reactivated, the statistical usage data is collected again and transferred to Siemens. The collection and transmission of statistical usage data only takes place in the corresponding version of the TIA Portal. Data from other TIA Portal installations with disabled setting will not be collected and transferred after reactivation. |  |

#### Exporting and importing settings

You can export settings of the TIA Portal to a file. When you import the settings, they are applied together. This function makes it possible to send settings to other users. Likewise, system settings can be specified centrally, to provide a unified environment for multiple engineering PCs. The TIA Portal searches for a initialization file at startup. When a initialization file is found, the settings saved therein are automatically applied after rebooting. If a more recent version of this file is available, the local settings are updated.

You can export all settings, for example:

- Language of the user interface
- Path of the company libraries
- Path of the user-defined documentation
- Window layouts
- Profiles of the hardware catalog

The initialization files are version-specific and can be imported automatically within a product version, but not in other versions of the product.

#### Exporting settings

You can export settings of the TIA Portal to a file, for example, to back them up or to share them with other users.

##### Procedure

1. Select the "Settings" command in the "Options" menu.

   The settings of the TIA Portal are displayed.
2. Select the "General" entry in the area navigation.
3. In the "Import/export settings" area, click on the "Export settings ..." button.

   The "Export settings" dialog opens.
4. Select the settings for the export.

   - Click "Select all" to select all the settings for the export.
   - Click "Deselect all" to then select individual settings for the export. It all the nested settings are selected as well.
5. If you want to accept the settings selected from previous export operations, for example for a different use of the initialization files, select the check box "Use selection from a previous export file" and enter the file in the box.
6. Enter a file name for the initialization file in the "File name" box.
7. Click "Export".

   The settings are exported to a file with the extension ".tps19".

**Note**

**Empty entry fields in the export file**

Entry fields, such as "Central settings", are available in the settings. If you do not fill out one of these entry fields and nevertheless select the initialization for the export, an empty initialization will be exported. If the subsequently import the initialization file, the corresponding local settings are overwritten by the empty contents of important initialization file.

If you do not want these fields to be deleted by an import, disable the non-filled entry fields during export of the initialization file.

#### Generate configuration for predefined settings

You can predefine program settings from a settings file that is automatically imported when the TIA Portal starts up. This gives you the option to centrally define the settings of the TIA Portal throughout the company. You can predefine the setting file as optional or mandatory. To do this, define the path to the settings file and, if necessary, the write protection to the central settings in an XML configuration file.

##### Requirement

- There is a setting file with the extension ".tps19".

##### Generate configuration file

1. Navigate to the following directory on the computer:

   C:\ProgramData\Siemens\Automation\Portal V19\CorporateSettings\
2. Open the "CorporateSettings.xml" configuration file with a text editor.

   If the configuration file does not exist, create an XML file with the contents specified below. Use the coding "UTF-8".
3. In the element "PredefinedCentralSettingsPaths", specify whether the options in the "Central settings" area are selectable by the TIA Portal user or write-protected:

   - To make the central settings write-protected, enter the value "true" for the attribute "IsReadOnly".
   - To make the central settings selectable, enter the value "false" for the attribute "IsReadOnly".
4. In the "Item" element, insert the complete path to the setting file with the extension ".tps19".
5. Save the file named "CorporateSettings.xml" in the following directory on the computer:

   C:\ProgramData\Siemens\Automation\Portal V19\CorporateSettings\
6. Ensure that the configuration file is available for the installation of the TIA Portal on the target computers.

##### Contents of the configuration file

The XML configuration file must have the following structure:

XML

&lt;?xml version="1.0" encoding="utf-8"?&gt;

&lt;Document&gt;

&lt;Settings.Settings ID="0"&gt;

&lt;ObjectList&gt;

&lt;Settings.General ID="1" AggregationName="General"&gt;

&lt;AttributeList&gt;

&lt;PredefinedCentralSettingsPaths IsReadOnly="true"&gt;

&lt;!-- Example for the path --&gt;

&lt;Item&gt;D:\CentralSettings\Settings.tps19&lt;/Item&gt;

&lt;/PredefinedCentralSettingsPaths&gt;

&lt;/AttributeList&gt;

&lt;/Settings.General&gt;

&lt;/ObjectList&gt;

&lt;/Settings.Settings&gt;

&lt;/Document&gt;

You define the content of the XML configuration file within the "Settings.General" element.

If the "CorporateSettings.xml" file already exists, add the "Settings.General" element within the "ObjectList" element.

##### Assigning user-defined keyboard shortcuts

You can set your own keyboard shortcuts for operating the TIA Portal.

###### Procedure

To assign user-defined keyboard shortcuts, follow these steps:

1. Select the "Settings" command in the "Options" menu.

   The settings of the TIA Portal are displayed.
2. Open the "Keyboard shortcuts" entry in the area navigation.

   You can see a table with an overview of all keyboard shortcuts, which are valid for the currently installed products.
3. Click on a row in the table.

   A button appears in the "User-defined keyboard shortcuts" cell.
4. Click on the button.

   The "Edit user-defined keyboard shortcuts" dialog opens. In the upper part of the dialog you can see the currently selected action, the editor and the currently assigned keyboard shortcut.
5. Select the "Assign new keyboard shortcuts" option.

   The option is set by default.
6. Open the drop-down list under the "Assign new keyboard shortcuts" option.

   The available keyboard shortcuts, not used in TIA Portal, are displayed.
7. Select a keyboard shortcut.
8. Click "OK" to apply your selection and close the dialog box.
9. Restart the TIA Portal in order to apply the assigned keyboard shortcut.

   The assigned keyboard shortcut is no longer available for new user-defined keyboard shortcuts. The previously used keyboard shortcut is available for new, user-defined keyboard shortcuts.

   User-defined keyboard shortcuts overwrite default keyboard shortcuts.

##### Deleting user-defined keyboard shortcuts

###### Procedure

To delete user-defined keyboard shortcuts, follow these steps:

   The settings of the TIA Portal are displayed.
2. Open the "Keyboard shortcuts" entry in the area navigation.

   You can see a table with an overview of all keyboard shortcuts, which are valid for the currently installed products.
3. Click on a row in the table.

   A button appears in the "User-defined keyboard shortcuts" cell.
4. Click on the button.

   The "Edit user-defined keyboard shortcuts" dialog opens. In the upper part of the dialog you can see the currently selected action, the editor and the currently assigned keyboard shortcut.
5. Select the "Remove keyboard shortcut" option.
6. Click "OK" to apply your selection and close the dialog box.
7. Restart the TIA Portal in order to apply the deletion of the keyboard shortcut.

   The deleted keyboard shortcut is again available for new, user-defined keyboard shortcuts.

#### Influence of user rights

##### Restrictions when user rights are limited

The software provides several functions that require direct access to the hardware of the programming device / PC and therefore also to the installed operating system. #### Basics of searching

##### Introduction

You have the option to search the information system for terms. You are provided a search area for this.

##### Search options

You can search for the following text:

- Search for a word:

  `<Search term>`

  This finds all help pages that contain the search term exactly or in a slightly changed form.
- Search for several words:

  `<Search_term_1 Search_term_2 ... Search_term_n>`

  This finds all help pages that contain all search terms exactly or in a slightly changed form. Neither the order nor the distance between the search terms plays a role here.
- Search for phrases

  `<"Search_term_1 Search_term_2 ... Search_term_n">`

  This finds all help pages that contain all search terms in the exact order specified. To find a phrase, place your search expression in quotation marks.

You can further specify your search with additional criteria:

- Boolean search with "AND":

  `<Search_term AND Search_term>`

  `<Search_term + Search_term`&gt;

  You can link your search terms using the Boolean "AND" operator. This finds all help pages that contain all search terms exactly or in a slightly changed form.

  Alternatively, you can enter multiple search terms even without the operators "AND" or "+". This too finds all help pages that contain all search terms.
- Boolean search with "OR":

  `<Search_term OR Search_term>`

  `<Search_term – Search_term`&gt;

  You can link your search terms using the Boolean "OR" operator. This finds all help pages that contain one of the search terms exactly or in a slightly changed form.
- Negate search with "NOT"

  `<Search_term_1 NOT Search_term_2>`

  `<Search_term_1 ! Searchterm_2>`

  You can negate a search term with "NOT". You must enter at least one additional search term for this purpose, however. This finds all help pages in which "Search_term_1" occurs. Pages in which "Search_term_2" occurs are not included in the hits, since this search term has been negated.
- Fuzzy search with "~"

  `<Search~>`

  You can place a tilde ("~") after the search term. This finds all help pages in which the search term appears with approximately this spelling.
- Distance search with "~&lt;proximity&gt;"

  `<"Search_term_1 Search_term_2" ~proximity>`

  The proximity search allows you to find help pages where two search terms are no further apart than the defined distance. Enter the proximity as a positive integer. Enter the two search terms in quotes. There should be no gap between the tilde ("~") and the space.

##### Search filter

You can restrict the search by specifying search filters:

- If you select a device as a filter criterion, the keyword is only searched for in the pages or sections that apply to the specified device.
- If you select an area as a filter criterion, the search term is searched either only in the software documentation or only in the integrated hardware manuals.

The two search filters can also be combined.

##### Extended search

You can also use the extended search to search the information system. It offers the same search options as the default search. However, you also obtain direct information about the possible selections for search options.

A Boolean search with "OR" finds all help pages that contain one or more of the specified search terms exactly or in a slightly changed form.

- Search for `<block OR function>`

  This finds all pages containing the terms "block", "blocks" and/or "function", "functions" or similar related terms.
- Search for `<block OR function OR function block>`

  This finds all pages containing the terms "block", "blocks" and/or "function", "functions" and/or "function block", "function blocks" or similar related terms.

  This finds all pages containing the terms "alarms", "alarm" and "CPU" but not the word "GRAPH".
- Search for `<alarm NOT CPU NOT GRAPH>`

  All pages containing the term "message" are found but not the word "GRAPH" and not the word "CPU".

A fuzzy search finds all help pages in which the search term appears with approximately this spelling.

  This finds all pages in which the terms "trend" and "measurement" are no more than 10 words apart.

#### Basics for search in the project

##### Introduction

You can search the open project for a specific text (search term). The search term can be one word, several words or parts of words. Only spaces are considered as word limits. You can use search terms that are up to 255 characters long. You can create any complex search terms using wildcards. You can also limit the search to specific areas, objects or object properties. Uppercase and lowercase letters are not taken into account in the search.

For the search, an index of the texts of the editors which can be searched in the project is built up automatically in the background. The search index is updated with each change. You cannot start the search before indexing is complete. You can generate the search index again at any time.

Note that the search index is saved with the project. This may increase the file size of the project. If you want to pass on the project, you can reduce the file size by archiving or minimizing the project. The search index is removed in this case. You can also completely disable the search in the project if necessary.

In the case of changes to STEP 7 objects, e.g. name changes of tags, PLC data types or blocks, you must update the point of use of the these object before any search in the project in order that the search can take these changes into account. To do this you can, for example, recompile your program.

The search only includes the texts in the set editing language. The search index is generated again when you change the editing language. This may take some time, depending on the number of texts in the project. Also note that the search in the project does not take into account know-how-protected blocks.

The search editor helps you with your search. In it, you can specify search options and start the search. After the search is completed, the search editor lists the objects which contain the search text. You can open the individual objects directly from the search editor. If the objects have viewable properties, you can display them in the Inspector window.

##### Search options

The following options are available for the search:

- Limit search:

  - Search in: You can select the sections of the project that are to be searched.
  - Find exact match: Finds exactly the text you have entered. The search term is searched as individual word or as phrase. You cannot use wildcards in the search term when you use this option, for example, when you enter "*" or "?" the search is exactly for these characters.
  - Limit search: You can limit the search to specific object properties or object types.
- Use "*" as wildcard for any number of characters and "?" as wildcard for exactly one character within a word:

  When you use the "*" or "?" wildcards in a search term, you will receive whole words that match this search term as hits. When you enter "Motor*start", for example, you will receive hits such as "Motor starter" or "Motor1_Start" but not "Starting motors" or "Start motor1".

  An implicit wildcard search is always conducted. If you enter "Motor" as search term, for example, the search also finds the texts "Motor_ON", "Motor starter" and "123_Motor". This corresponds to the explicit wildcard search with the search term "*Motor*". You can deactivate the implicit wildcard search by enabling the option "Find exact match".

##### Replace texts in the project

You have the option to replace individual texts in the project. To do this, you can open the location immediately after a search by double-clicking on the hit list and editing the text. Once it is changed, the text is automatically indexed again. If you then want to open the next match, you receive a message that offers you the chance to refresh the hit list. In this way, you can replace a text in the entire project until the hit list no longer shows results.

##### Examples for use of property and object filters

- If you do not activate any filters, the search term is searched in all objects and properties that can be searched within the set search area.
- If you activate filters, only the selected properties or objects are searched.

Examples for use of property and object filters:

- Search for an object with the name "Motor1":

  Enter the search term and activate the filter "Name". All objects which include "Motor1" in their name are found.
- Search for an object which includes the text "Motor1" either in its name or in a comment:

  Enter the search term and activate the filters "Name" and "Comment". All objects which include "Motor1" in their name or in the comment are found.
- Search for a block with the name "Motor1":

  Enter the search term and activate the filters "Name" and "Block". All blocks which include "Motor1" in their name are found.
- Search for a block which includes the text "Motor1" either in its name or in a comment:

  Enter the search term and activate the filters "Name", "Comment" and "Block". All blocks which include "Motor1" in their name or in the comment are found.
- Search for a block or a technology object with the name "Motor1":

  Enter the search term and activate the filters "Name", "Block" and "Technology object". All blocks or technology objects which include "Motor1" in their name are found.

When entering the search term, note the following examples for searching for a term or several terms.

##### Examples for the search for one term

When your search entry consists of only one term and the option "Find exact match" is disabled, the search is always an implicit wildcard search. The system adds the wildcard "*" in front of and behind the search term. You can insert wildcards in the search term yourself to conduct an explicit wildcard search. If you want to find exactly the term you have entered, select the option "Find exact match". Uppercase and lowercase letters are not taken into account in the search.

Examples for the search for one term:

- Search for "motor", "Find exact match" option is disabled

  An implicit wildcard search is conducted. The search term is interpreted as "*motor*" and all terms which include "motor" are found, for example, "motor", "motor_1", "Motorstart" or "Start_Motor".
- Search for "m*tor" (explicit wild card search), "Find exact match" option is disabled

  The search term is interpreted as "*m*tor*" in this wildcard search. This means terms, such as "motor", "motor_1", "Start_Motor", "GLOB.Mode.motor[1]" or "Mo_tor" are also found.
- Search for "motor", "Find exact match" option is enabled

  No wildcards are used for the search, which means only the terms "motor" and "Motor" are found.
- Search for "m*tor" (explicit wild card search), "Find exact match" option is enabled

  The character "*" is no longer a wildcard and is interpreted as a regular character. The search only finds the character strings "m*tor" and "M*tor" but not, for example, the terms "motor", "", "Motormotor_1".

Application examples:

- If you want to determine from where the "GLOB_DB.MODE.FG[6].IN.ERROR_DROP_START" element of a data block is accessed, you can use the cross-reference list. If access takes place by indirect addressing (e.g. with "#GLOB.MODE.FG[#FgNr].IN.ERROR_DROP_START := TRUE;"), however, the cross-reference list does not return any results. Use a wildcard search for "MODE.FG[*].IN.ERROR_DROP_START" across the entire project in this case.
- You want to find all instances where "ERROR_DROP_START" is used. Perform a search for "ERROR_DROP_START" across the entire project. The search also finds all "ERROR_DROP_START" elements within the "FG[]" array.

##### Examples for the search for multiple terms

When you enter a search term which consists of multiple terms and the "Find exact match" option is disabled, you will find those objects which include all these terms. The texts can appear in any order. If you want the terms to appear exactly as you have entered them in an object, select the "Find exact match" option. Uppercase and lowercase letters are not taken into account in the search.

Examples for the search for multiple terms:

- Search for "motor start", "Find exact match" option is disabled

  All objects which include "motor" and "start" are found. It does not matter how far apart they are from each other or in which order they appear. Possible hits therefore include: "Motor start", "start my motor", "my motor is in start mode"

  > **Note**
  >
  > The texts of a column are treated as a text object in data blocks and in block interfaces. Therefore, the search also returns data blocks and blocks in which the terms occur distributed over different lines.
- Search for "motor start*", "Find exact match" option is disabled

  As with the search for a word, an implicit wildcard search is always conducted for all words with a multiple word search. This means it is not necessary to explicitly enter a wildcard character. "motor start*" is interpreted as "*motor* *start*" and the same hits are found as with "motor start".
- Search for "motor start", "Find exact match" option is enabled

  When the option "Find exact match" is enabled, only objects which include both terms one after the other as a phrase are found. The search finds the search term, for example, in objects which include the texts "push motor start button" or "motor start on". No hits are returned for "start motor", "motorstart", "start my motor", "my motor is in start mode".
- Search for "motor start*", "Find exact match" option is enabled

  The character "*" is no longer a wildcard and is interpreted as a regular character. The search only finds objects in which the character string "motor start*" appears in uppercase or lowercase letters. "motor starting", "start my motor", "motor start", "Motor Start", "motor start_mode" do not result in any hits.

Application examples:

- You want to find out where you have used the function "Setbit" to set the "MotorOn" tag. To do so, conduct a search for "setbit motoron".
- You want to find a specific network comment but you only know that it includes the words "CPU" and "Cycle". To do so, start a search for "CPU cycle".

## Undoing and redoing actions

- [Undoing an action](#undoing-an-action)

### Basics of undoing and redoing actions

#### Function

You can undo performed actions at any time. For this purpose, every action you perform is saved in an action stack. When undoing actions, the stack is processed from top to bottom. In other words, if you undo an action that lies further down in the stack, all actions located above it in the stack will also be undone automatically.

You can redo previously undone actions until you execute a new action. Once you execute a new action, it is no longer possible to redo previously undone actions.

#### Particularities for undoing

There are a few actions that empty the action stack. You cannot undo these actions or the actions performed before these actions. The following actions empty the action stack:

- Saving
- Project management (creating a new project, opening project, closing a project, deleting a project)
- Downloading the project to an HMI device
- Deleting unused GSD files
- Establishing an online connection

Keep in mind that the action stack is emptied every time you establish an online connection. This means that the actions executed offline cannot be undone once the online connection is established. You can undo all actions which you subsequently execute online, even after the online connection has been terminated.

#### Displaying the action stack

The "Undo" button in the toolbar is enabled as soon as you perform an action that can be undone. This button is split; you can use the arrow down portion to open a drop-down list. containing all actions of the action stack that you can undo. If you had performed actions in an editor other than the currently displayed editor, the corresponding editor is also displayed as a subheading. This allows you to always identify the point at which the undo operation will be applied. The subheadings are removed from the list when the editor responsible can no longer undo actions.

Actions you have undone are entered in the action stack from where they can be redone. Here, you can redo actions you have undone. The display of actions you can redo it is analogous to the display of the actions that you can undo.

#### Example of undoing actions

The figure below shows how actions performed in various editors and tables are undone:

![Example of undoing actions](images/20258687371_DV_resource.Stream@PNG-en-US.png)

In this example, you cannot undo actions 1 to 3 because the project was saved. You can undo actions 4 to 10 in the order indicated by the direction of the arrow. In other words, you must undo action 10 first. Once you have undone action 8, you cannot then undo action 5. You must first undo actions 7 and 6. As the final step in the sequence, you can then undo action 4. You also have the option of undoing several actions in a single step by undoing an action located further down in the action stack. All actions located above it in the stack will be undone automatically.

The same principle also applies to redoing of actions.

   The editor in which the action was performed is displayed and the action is undone.

   The chosen action and all actions in the stack located above the chosen action are undone. If the undo operation requires an editor containing a protected object to be opened, you must enter the password for the object.
3. Enter the required passwords, if necessary.

   The editors in which the actions were performed are displayed and the actions are undone.

#### Introduction to programming Add-Ins

##### Introduction

Add-Ins enable you to extend TIA Portal to include your own functions. You can either integrate these functions into the user interface of the TIA Portal via shortcut menus or extend certain workflows with them. You can program shortcut menus for the following areas:

- Project tree
- "Libraries" task card: Project library and global libraries
- Hardware and network editor
- Version Control Interface workspace

You can define yourself which element should be visible and clickable or define the conditions for this.

You can use the following elements in your shortcut menu:

- Submenus
- Simple clickable buttons
- Buttons with check boxes
- Buttons with option buttons

For information on workflows that you can extend using Add-Ins, see "[Basic information on Add-Ins](#basics-of-add-ins)".

You create the program code for an Add-In with C# and the TIA Portal Openness API. To do this, create a class library from which an .addin file can be generated later. The descriptions and examples in this help document use Microsoft Visual Studio for this. If you use a different development environment, refer to the documentation of this development environment if you have any questions.

##### Requirements

To program an Add-In for TIA Portal, you should have the following skills:

- Knowledge of C# and .NET
- Knowledge of TIA Portal Openness

##### Overview of program creation

To program an Add-In, first generate a C# project and insert the necessary references. A simple Add-In consists of at least the following two classes:

- View Provider: In this class, you define the areas in which the shortcut menu is to be displayed. For each area you have to program a separate provider, i.e. a separate class:

  - ProjectTreeAddInProvider: Project tree
  - GlobalLibraryTreeAddInProvider: Global libraries
  - ProjectLibraryTreeAddInProvider: Project library
  - DevicesAndNetworksAddInProvider: Hardware and network editor
  - VciEditorAddInProvider: Version Control Interface workspace
- Shortcut menu: In this class, you define the shortcut menu with its entries. If your Add-In is not very extensive, you can also create your program code in this class.

You create the executable program code for your button next with C# and the TIA Portal Openness API. You can also configure the following information in a configuration file in XML format:

- Information on the product: Name, ID, version
- Permissions
- Information on certificates
- Display of the Add-In in a multiuser session

When you have finished programming your Add-In, the resulting dll file must be converted into a file with the file extension "addin". For this, you use the supplied program "Siemens.Engineering.AddIn.Publisher.exe" and your configuration file. Copy the addin file to the installation directory of TIA Portal. After this, you can activate the Add-In in TIA Portal and use it.

> **Note**
>
> Note the following:
>
> - Note that the Add-In is unloaded after execution, and therefore the instance executed can no longer be used. This means that tasks running longer in a background thread are canceled. Start a new process for longer running tasks.
> - Closing a project via a TIA Portal Add-In is not recommended.

##### Permissions

In the configuration file, you can configure the following permissions for your Add-In in the "RequiredPermissions" element:

- TIAPermissions: Read and write permissions for TIA Portal
- SecurityPermissions: Security-related permissions, e.g. starting processes
- UnrestrictedPermissions: Unrestricted access to the TIA Portal

You cannot use "SecurityPermissions" and "UnrestrictedPermissions" together. You either assign specific rights with "SecurityPermissions" or allow unrestricted access with "UnrestrictedPermissions". For example, unrestricted access is required if you want to access Microsoft Interop APIs or use specific references.

Note, however, that Add-Ins with unrestricted access have all the rights of the logged-in user. As a result, they have more authorizations and can be potentially dangerous and also enable the execution of malicious code. Therefore, you must specify the reason for unrestricted access in the configuration file. This reason is displayed in the TIA Portal in the "Add-Ins" task card within the "Details" palette.

##### TIA Portal Add-In Development Tools

You can either create the C# project and the necessary classes and methods manually using the descriptions below or use the TIA Portal Add-In Development Tools. The TIA Portal Add-In Development Tools make the following tasks easier for you:

- Creating the C# project

  You can use a project template to create the C# project for your Add-In. This way the project already contains important classes and methods.
- Creating program code

  You can use templates to create a program framework for the desired Add-In type. Then you program the desired functions.
- Compiling and debugging

  The TIA Portal Add-In Development Tools allow you to easily compile and debug your program code.

Use the TIA Portal Add-In Development Tools in Microsoft Visual Studio 2019 or 2022 or in Visual Studio Code.

You can find more information about the TIA Portal Add-In Development Tools and the required installation files on the DVD 2 of the TIA Portal Installation Package.

#### Programming the View Provider

You use the View Provider to define the TIA Portal area in which the shortcut menu with your Add-In is displayed. Create a separate class for each provider type and create the code for the respective view. You can program the following View Providers:

- "ProjectTreeAddInProvider":

  Shows the shortcut menu in the project tree.
- "GlobalLibraryTreeAddInProvider":

  Shows the shortcut menu in global libraries.
- "ProjectLibraryTreeAddInProvider":

  Shows the shortcut menu in the project library.
- "DevicesAndNetworksAddInProvider":

  Shows the shortcut menu in the hardware and network editor. Note, however, that the shortcut menu cannot be displayed in the following areas of the hardware and network editor:

  - "I/O communication" in the network view
  - "Connections" table in the network view
- "VciAddInProvider":

  Shows the shortcut menu in a Version Control Interface workspace.

##### Requirement

A C# project exists for creating a class library.

##### Procedure

To program a View Provider, follow these steps:

1. Create a new class in your project and give it a meaningful name, for example, "ProjectTreeProvider.cs", when you create a shortcut menu for the project tree.
2. Add the following two using directives to the class:

   - `using Siemens.Engineering.AddIn;`
   - `using Siemens.Engineering.AddIn.Menu;`
3. Declare your class as "public" and make the class a subclass of your desired provider type, e.g. for "ProjectTreeAddInProvider":

   `public class ProjectTreeProvider : ProjectTreeAddInProvider`
4. Enter "override" in the new class as text and select the entry "GetContextMenuAddIns" in the displayed drop-down menu.

   The method is supplemented to include all relevant components.
5. Remove the row "return base.GetContextMenuAddIns();" and enter the following code instead:

   `yield return new`
    `<Your_`
   `ContextMenuName`
   `_Classname>("<Text of the shortcut menu>");`

   If you program your shortcut menu in a "AddInContexMenu" class and the entry in the shortcut menu is to be called "My AddIn", the line would look as follows:

   `yield return new AddInContexMenu("My AddIn");`

##### Example: "ProjectTreeAddinProvider"

The following example shows the complete program code for a View Provider for a shortcut menu in the project tree:

using Siemens.Engineering.AddIn;

using Siemens.Engineering.AddIn.Menu;

namespace Addin_Test

{

public class ProjectTreeProvider: ProjectTreeAddInProvider

{

protected override IEnumerable&lt;ContextMenuAddIn&gt; GetContextMenuAddIns()

{

yield return new TestContextMenu("Test");

}

}

}

"ProjectTreeProvider" is the name of the class.

   `}`

   `protected override void BuildContextMenuItems(`ContextMenuAddInRoot `addInRootSubmenu)`

#### Creating the program code of the Add-In

You create the program code for your Add-In with C# and the TIA Portal Openness API. Make sure that all necessary references and using directives exist. You can basically use the complete scope of the TIA Portal Openness API for your Add-In. However, there are some special features for the following two functions that you keep in mind:

- Starting another process using the Add-In:

  In addition to the program code for the connection, starting a new process must be permitted in the configuration file.
- Creating a connection to another TIA Portal instance:

  You need the following reference to the assembly "Siemens.Engineering.AddIn.dll" to create Add-Ins. An additional reference to the assembly "Siemens.Engineering.dll" is required for starting another TIA Portal instance. However, the two assemblies contain some identical fully qualified type names. You can use external assembly aliases to inform the compiler about the assembly from which a type is to be used. This process is similar to a namespace. Without aliases you will receive a compile error.

##### Starting another process using the Add-In:

To start another process via an Add-In, follow these steps:

1. Open the configuration file.
2. Add the following code as the content of the "SecurityPermissions" tag:

   `<Siemens.Engineering.AddIn.Permissions.ProcessStartPermission>`

   `<Comment>`
    `Any comment`
   `</Comment>`

   `</Siemens.Engineering.AddIn.Permissions.ProcessStartPermission>`
3. If you have not yet created a reference to the "Siemens.Engineering.AddIn.Utilities.dll" file, add this reference now.
4. Create the desired program code for starting a process.

##### Creating a connection to another TIA Portal instance

To set up a connection to another TIA Portal instance, follow these steps:

1. Open your C# project.
2. Select the reference to the file "Siemens.Engineering.dll" and open the properties.
3. In the "Aliases" field, enter an alias name, e.g. "OpnsAPI". This can be any text.
4. Select the reference to the file "Siemens.Engineering.AddIn.dll" and open the properties.
5. In the "Aliases" field, enter an alias name, e.g. "AddInAPI". This can be any text.
6. Enter the following two alias directives in the directives:

   - `extern alias`
      `<Alias name for the Openness API>`
   - `extern alias`
      `<Alias name for the` 
     `Add-In`
      `API>`
7. Use the alias names as prefix when you program the connection to a new TIA Portal instance. Example:

   `OpnsApi::Siemens.Engineering.TiaPortal` newTiaPortalInstance `= new OpnsApi::Siemens.Engineering.TiaPortal();`

When you are trying to set up a connection to a running TIA Portal via the TIA Portal Openness, the TIA Portal will prompt you to accept the connection or reject it if you have not created a corresponding whitelist entry. For more information, refer to the section "TIA Portal Openness firewall" in the Openness documentation. For security reasons, do not click on "Yes to all".

##### Introduction to message APIs

Using Add-In APIs, you have the option of programming messages in your Add-Ins that are displayed to the user during runtime. These messages do not differ visually from other TIA Portal messages. You can integrate your Add-In seamlessly into the TIA Portal.

There are different APIs for the different types of messages. You can incorporate these APIs into your program code without further installations. The following APIs are available:

- Progress functions

You have the option to use the feedback service of the TIA Portal in order to display messages to the user of your Add-In. This allows you to provide detailed information in the event of an error, for example. The location at which these messages are displayed depend on the type of the Add-In:

### Mass rollout of Corporate Add-Ins

This section contains information on the following topics:

- [Basics of the mass rollout of Corporate Add-Ins](#basics-of-the-mass-rollout-of-corporate-add-ins)
- [Setting up mass rollout for Corporate Add-Ins](#setting-up-mass-rollout-for-corporate-add-ins)
- [Disable and enable mass rollout dialog](#disable-and-enable-mass-rollout-dialog)

#### Basics of the mass rollout of Corporate Add-Ins

##### Introduction

Add-Ins that are used throughout a company are called "Corporate Add-Ins". To distribute these Corporate Add-Ins as efficiently as possible in the company, a mass rollout can be performed.

The mass rollout works as follows:

- The TIA Portal is installed together with the Windows "Siemens Add-In Rollout Service" on the computers of the users. However, the service is not started during this process.
- An "Add-In Administrator" defines a directory of their choice in the network as the "Corporate Add-Ins network directory" and creates a specific substructure in it. In this directory, the administrator places released Add-Ins, which are to be distributed as Corporate Add-Ins.
- An "Add-In Rollout Manager" starts the Windows "Siemens Add-In Rollout Service" on the computers of the users. This can also be done simultaneously for several computers using a script.
- The service now checks the network directory and the "CorporateAddIns" folder in the TIA Portal installation directory for differences at a specified time interval and performs synchronization.

If the content of the "CorporateAddIns" folder was changed via synchronization, a dialog informs the user about this. This dialog shows which Corporate Add-Ins have been added, replaced or removed. The display of the dialog can be disabled in the settings at any time. In this case, the corresponding information is displayed in the Inspector window.

##### Required permissions

In order for the mass rollout of Corporate Add-Ins to be performed, the following permissions are required:

- "Add-In Administrator" role:

  - Needs write permission for the Corporate Add-Ins network directory.
  - Does not need write permissions on the computers of the users
- "Add-In Rollout Manager" role:

  Requires administrator rights on the computers of the users to start the Windows "Siemens Add-In Rollout Service" or possibly configure the service with the credentials of the user.

#### Setting up mass rollout for Corporate Add-Ins

For the mass rollout for Corporate Add-Ins to work, the Add-In administrator must set up a Corporate Add-Ins network drive and place the appropriate Add-Ins in it. The Add-In Rollout Manager takes care of running the Windows "Siemens Add-In Rollout Service" on the user computers where these Corporate Add-Ins are to be used. The Add-In Rollout Manager can start the service on one computer or simultaneously on multiple computers. The manager can stop and restart a running service on multiple computers simultaneously.

##### Setting up the Corporate Add-Ins network directory

To set up the Corporate Add-Ins network directory as an Add-In administrator, proceed as follows:

1. Create a new folder in the desired network directory.
2. Name the "TIAP&lt;version&gt;" folder, where "&lt;version&gt;" is the version of the TIA Portal for which the Add-Ins are to be used. For example, for TIA Portal V18, the folder name is "TIAP18".

   Note that the Windows service for the mass rollout only considers directories that start with "TIAP".
3. You can then store all Add-Ins you want to distribute within your company in this folder. The mass rollout is executed once the Windows service has been started on the appropriate computers.

##### Optional: Configure Windows service

If the Corporate Add-Ins network directory is configured so that everyone can access it, this configuration step is not necessary. If this is not possible, the Windows service must be configured with a user account that has write access to the "CorporateAddIns" directory in the TIA Portal installation directory and read access to the network directory.

To configure the Windows service of a user as the Add-In Rollout Manager, proceed as follows:

1. Open a Windows command prompt (cmd) either locally on the user computer or on your computer.
2. If you are accessing the user computer locally, enter the following command:

   `sc.exe config "Siemens Add-In Rollout Service" obj= "domain\userid" password= "*****"`

   Replace the domain, user ID and password with the correct values.
3. If you perform the configuration remotely, enter the following command:

   `sc.exe \\[Hostname] config "Siemens Add-In Rollout Service" obj= "domain\userid" password= "*****"`

   Replace the hostname, domain, user ID and password with the correct values.

> **Note**
>
> **Password change**
>
> If you later want to change the password from the configured user account or the password has expired and needs to be renewed, you should first stop the service on all computers that use this user account for access. Then change the password and restart the service on all computers.

##### Starting Windows service on a user computer

To configure the Windows service on a user computer as Add-In Rollout Manager, proceed as follows:

1. Open a Windows command prompt (cmd) either locally on the user computer or on your computer.
2. If you are accessing the user computer locally, enter the following command:

   `sc.exe start "Siemens Add-In Rollout Service" "\\[`Corporate Add-Ins network directory`]" "[Poll Time in minutes]"`

   Replace the network path and the time interval (Poll Time) with the correct values. The time interval is the interval at which synchronization between the Corporate Add-Ins network directory and the user computer occurs. Enter a time interval equal to or greater than 5 minutes.
3. If you perform the configuration remotely, enter the following command:

   `sc.exe \\[Hostname] start "Siemens Add-In Rollout Service" "\\[`Corporate Add-Ins network directory`]" "[Poll Time in minutes]"`

   Replace the host name, network path and time interval (Poll Time) with the correct values. The time interval is the interval at which synchronization between the Corporate Add-Ins network directory and the user computer occurs. Enter a time interval equal to or greater than 5 minutes.

The service has been started on the user computer and will now synchronize the Add-Ins on the user computer with the Add-Ins in the Corporate Add-Ins network directory at the specified time interval.

##### Starting Windows service on multiple user computers simultaneously

To start the Windows service simultaneously on multiple user computers as an Add-In Rollout Manager via a Powershell script, proceed as follows:

1. Copy the following PowerShell script into a text editor or the Windows PowerShell ISE.
2. Replace "&lt;Hostnames text file path&gt;" with the path to a text file containing the user computers which you want to start the service.
3. Replace "&lt;Corporate Add-Ins network directory&gt;" with the path to the network directory where the Corporate Add-Ins are managed.
4. Adapt the time interval if required. In the example below, 5 minutes is set.
5. If the above optional configuration step is required to configure the Windows service, replace "TestUser" and "TestPassword" with the correct user data. Otherwise, remove the entire line "Invoke-Command { sc.exe \\$name config "Siemens Add-In Rollout Service" obj= "domain\TestUser" password="TestPassword" }".
6. Run the script.

Example: PowerShell script to start the Windows service on multiple computers

$targetHostname = Get-Content -Path '&lt;Hostnames text file path&gt;' ;  
 foreach ($name in $targetHostname)   
 {  
 $MassRollOutService = Get-Service -name "Siemens Add-In Rollout Service" -ComputerName $name;

Invoke-Command { sc.exe \\$name config "Siemens Add-In Rollout Service" obj= "domain\TestUser" password="TestPassword" }

$MassRollOutService.Start(@('&lt;Corporate Add-Ins network directory&gt;','[Poll Time in minutes]'))  
 $MassRollOutService.WaitForStatus('Running', '00:00:30')   
  
 if($MassRollOutService.status -eq "Running")  
 {   
 Write-Output "Service started running on" $name;  
 }  
 else  
 {  
 Write-Output "Service did not start on" $name;  
 }  
  
}

##### Troubleshooting when starting the Windows service

If the Windows service has been configured with a user account, the local policy "Log on as a service" is required on the computer where the service is to run. If the local policy has been overridden by the group policy, the service cannot be started and there is an entry in the log files under "C:\ProgramData\Siemens\Automation\AddInRollout". In this case, the Add-In Administrator must update the group policy of the domain and re-establish the "Log on as a service" right for the users of the service.

##### Stopping Windows service on multiple user computers simultaneously

To stop the Windows service simultaneously on multiple user computers as an Add-In Rollout Manager via a Powershell script, proceed as follows:

1. Copy the following PowerShell script into a text editor or the Windows PowerShell ISE.
2. Replace "&lt;Hostnames text file path&gt;" with the path to a text file that contains the user computers for which you want to stop the service.

3. Run the script.

Example: PowerShell script to restart the Windows service on multiple computers

$targetHostname = Get-Content -Path '&lt;Hostnames text file path&gt;' ;  
 foreach ($name in $targetHostname)   
 {  
 $MassRollOutService = Get-Service -name "Siemens Add-In Rollout Service" -ComputerName $name;  
 $MassRollOutService.Stop()  
 $MassRollOutService.Start()  
  
 $MassRollOutService.WaitForStatus('Running', '00:00:30')   
  
 if($MassRollOutService.status -eq "Running")  
 {   
 Write-Output "Service started running on" $name;  
 }  
 else  
 {  
 Write-Output "Service did not start on" $name;  
 }  
  
 }

##### Result

The mass rollout for Corporate Add-Ins has been set up. If errors occur when starting the Windows service or during synchronization, you can find details about them in the log files under "C:\ProgramData\Siemens\Automation\AddInRollout".

#### Disable and enable mass rollout dialog

If the Windows service for the mass rollout of Add-Ins has been started, the "CorporateAddIns" directory in the installation directory of the TIA Portal is synchronized with the Corporate Add-Ins network directory at a certain time interval. When changes are made, a dialog is displayed informing you which Add-Ins have been added, replaced or removed. You can disable and re-enable this dialog at any time via the TIA Portal settings or via the Openness API. If the dialog is disabled, you can obtain the corresponding information from the Inspector window.

##### Enabling and disabling the dialog via the TIA Portal settings

To enable or disable the dialog via the TIA Portal settings, proceed as follows:

1. Select the "Settings" command in the "Options" menu.

   The "Settings" window is displayed in the work area.
2. Select the "Add-Ins" group in the navigation area.
3. If you do not want to receive any additional notification, select the "Disable notification by popup window with mass rollout of Add-Ins" check box.
4. If you want to turn the additional notification again, clear the check box.

##### Disable and enable dialog via Openness API

To enable or disable the dialog via the Openness API, you can copy and modify the following code example:

`TiaPortalSetting rolloutPopupSetting = addInSettingsFolder.Settings.Find("DisableAddInMassRolloutPopup");`

`{`

`}`

`...`

### Enabling and disabling Add-Ins

#### Requirements

- TIA Portal Openness is installed.
- The package files with the file extension "*.addin" were saved to the "AddIns" folder in the TIA Portal installation directory.

#### Activate Add-Ins

To activate "Add-Ins", follow these steps:

1. Open the "Add-Ins" task card.
2. In the "Add-Ins" pane, navigate to the Add-In that you want to activate.

   The authorizations required for the selected Add-In are displayed in the "Details" pane.
3. In the shortcut menu of the Add-In or in the "Status" column, select the command "Activate".

   When the Add-In is a workflow Add-In, the "Configuration of the default Add-In" is displayed. In this case, continue with step 4.
4. In the "Default Add-In" column, select the Add-In from the drop-down list that is to be executed by default together with the corresponding workflow.

   When you close the dialog using the "Cancel" button, the Add-In is still activated but it will not be used as the default Add-In. You can define, change or remove the standard Add-In for a workflow at any time in the configuration view.

   See also: [Managing standard add-ins for workflows](#managing-default-add-ins-for-workflows)
5. Repeat the procedure for all Add-Ins that you want to activate.

   Activated Add-Ins can be identified by the green check mark in the "Status" column and can be used in the shortcut menus or together with the corresponding workflows.

#### Deactivating Add-Ins

To deactivate "Add-Ins", follow these steps:

1. Open the "Add-Ins" task card.
2. In the "Add-Ins" pane, navigate to the Add-In that you want to deactivate.
3. In the shortcut menu of the Add-In or in the "Status" column, select the command "Deactivate".
4. Repeat the procedure for all Add-Ins that you want to deactivate.

   You can identify deactivated Add-Ins by the blue X in the "Status" column. They can no longer be used in the shortcut menus or run as standard Add-In together with a workflow.

### Managing default Add-Ins for workflows

You can define which Add-In should be run with a workflow by default. This Add-In is then the default Add-In for this workflow. In the configuration view, you obtain an overview of the Add-Ins workflow and the assigned default Add-Ins in each case. You can either assign the default Add-In during activation or later via the configuration view. In addition, you can change or remove assignments in the configuration view.

The following figure shows the configuration view and the button in the "Add-Ins" task card via which you can open the configuration view:

![Figure](images/154602163339_DV_resource.Stream@PNG-en-US.png)

#### Procedure

Follow these steps to define, change or remove a default Add-In for a workflow:

1. Open the "Add-Ins" task card.
2. Click on the "Open configuration view for Add-Ins" button in the task card above the panes.

   The configuration view opens.
3. Select the row of the workflow for which you want to change the assignment of the default Add-Ins.
4. In the "Default Add-In" column, select the Add-In that is to run together with the corresponding workflow from the drop-down list. If you want to remove an assignment, select the entry "&lt;None&gt;".

### Displaying certificates

Add-Ins can optionally be signed with a certificate. Certificates are displayed in the "Details" pane of the "Add-Ins" task card.

The following certificate trust levels are displayed in the "Details" pane:

| Symbol | Meaning |
| --- | --- |
| ![Figure](images/127484593931_DV_resource.Stream@PNG-de-DE.png) | The certificate is evaluated by the system as "trusted". The package has not been modified. |
| ![Figure](images/127485792907_DV_resource.Stream@PNG-de-DE.png) | The certificate is unsigned, signed with an invalid certificate, or is invalid. |
| ![Figure](images/127485827083_DV_resource.Stream@PNG-de-DE.png) | The Add-In is signed with a revoked certificate, or the package has been modified. |

#### Procedure

To display a certificate, proceed as follows:

1. Open the "Add-Ins" task card.
2. Select an Add-In from the "Add-Ins" pane.
3. Change to the "Details" pane.

   The properties of the certificate are displayed in the following fields:

   - Trust level: Shows the trust level of the certificate.
   - Issuer: Shows the issuer of the certificate.
4. To display other properties of the certificate click the "View certificate" link in the "Details" pane.

   The other properties of the certificate are displayed in a dialog.

   If an Add-In is not signed with a certificate, these fields are empty. The dialog for displaying the certificate cannot be called up.

### Running Add-Ins

You only need to explicitly execute those Add-Ins that you have integrated into a shortcut menu. Add-Ins for workflows are executed together with the respective workflow.

#### Requirement

The Add-Ins have been activated the "Add-Ins" task card.

#### Procedure

To execute Add-Ins, follow these steps:

1. Select the object in whose context you want to run the Add-In.

   You can run Add-Ins in the project tree, in the "Devices &amp; networks" editor, and in the "Libraries" task card.
2. Open the shortcut menu.
3. In the shortcut menu under the "Add-Ins" entry, execute the command "&lt;Name of the Add-In&gt; &gt; &lt;Name of the Add-In&gt;".

> **Note**
>
> **Closing projects using Add-Ins**
>
> Do not use any actions to close projects in Add-Ins. Closing projects using Add-Ins can result in an unexpected program crash.
